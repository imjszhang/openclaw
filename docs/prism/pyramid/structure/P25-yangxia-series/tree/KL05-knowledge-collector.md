# KL05: 知识收集器是 AI 时代的"感官系统"——让龙虾能看、能记、能思考

> 所属视角：P01-P01-yangxia-series  
> 上层论点：养虾日记系列——OpenClaw 实践记录

## 核心命题

**知识收集器不是简单的"收藏工具"，而是 AI Agent 的感官系统**——它让 OpenClaw 能够：

- 👁️ **看**：抓取网页内容（微信公众号、知乎、小红书、即刻、X.com 等）
- 🧠 **记**：AI 自动总结（概要/摘要/推荐理由），结构化入库
- 🔗 **连**：与 Flomo、Prism 知识库、Memory-Core 打通，形成完整知识链路

**关键洞察**：没有感官系统的 AI 是"闭门造车"，知识收集器让龙虾真正"活"在信息洪流中。

## 支撑论点

### 5.1: 知识收集器的三层架构——从抓取到内化

**第一层：抓取（Input）**

- 支持平台：微信公众号、知乎、小红书、即刻、X.com、Reddit、Bilibili、YouTube、GitHub、通用网页
- 技术实现：`knowledge_collect` 工具 + `js-knowledge-collector` CLI（集成 JS-Eyes 浏览器插件）
- 核心能力：复用浏览器登录态，突破登录墙（即刻/知乎/小红书）

**第二层：处理（Process）**

- AI 自动总结：生成概要、摘要、推荐理由
- 结构化存储：Markdown 格式，包含原文链接、封面图、创建时间
- 标签系统：自动推荐标签，支持手动标注

**第三层：内化（Integrate）**

- Flomo 推送：速记层，快速检索
- Prism 知识库：分析层，结构化洞察
- Memory-Core：检索层，统一语义搜索

**关键问题**：

- 登录墙问题：即刻移动端页面需转换为 PC 版（`m.okjike.com` → `web.okjike.com`）
- Flomo 推送 Bug：需添加 `#概要` 标签，条件判断需优化
- 数据孤岛：Flomo 笔记在云端，Prism 无法直接分析，需定期导出回灌

### 5.3: 知识收集器的问题与修复——实战中的 Bug 追踪

**问题 1：即刻登录墙（2026-03-07）**

- 现象：`knowledge_collect` 无法抓取即刻移动端页面
- 根因：需要登录态，且移动端 URL 格式不同
- 修复：使用 `js-knowledge-collector` CLI 工具，集成 JS-Eyes 浏览器插件
- 代码修复：`browser-automation.js` 和 `scraper.js` 中添加 URL 转换逻辑
  - 移动 URL: `https://m.okjike.com/originalPosts/xxx`
  - PC URL: `https://web.okjike.com/originalPost/xxx`

**问题 2：Flomo 推送 Bug（2026-03-07 晚间）**

- 现象：传了 `flomo: true` 但不发送，或发送时没有 `#概要` 标签
- 根因：
  1. 条件限制：必须有 `summaryDir` 才发送
  2. 格式缺失：直接发送 `digest.txt`，未加标签前缀
- 临时方案：手动脚本补发（已补发 4 篇）
- 待修复：
  - `collector.js` - Flomo 发送前添加 `#概要\n\n` 前缀
  - `link-collector` SKILL.md - 即时入库流程读取 `config.json` 并传递 `flomo` 参数

**问题 3：ClawHub 博客同步权限错误（2026-03-08）**

- 现象：`clawhub-blog-auto-sync` cron 任务构建失败（EPERM 权限错误）
- 根因：目录被其他进程占用（编辑器、资源管理器等）
- 修复：关闭占用程序后重试，或手动 `git pull --rebase` 后再推送
- 状态：已解决，19:35 和 21:35 两次同步均正常完成

**经验教训**：

- 遇到登录墙平台（即刻/知乎/小红书），用 CLI 工具或手动复制内容入库
- Flomo 推送需默认开启，无需每次询问
- Cron 任务需处理目录占用和 Git 冲突的边界情况

### 5.4: 知识收集器的战略价值——构建个人知识操作系统

**长期愿景**：

```
输入：链接/想法/对话 → 自动分类入库
       ↓
处理：AI 总结 → 结构化 → 关联已有知识
       ↓
输出：按需检索 → 自动组装 → 多平台发布
       ↓
反馈：发布数据 → 回流优化标签/分类策略
```

**角色定位**：

- **Flomo** = 速记层（快速输入、移动端友好、标签检索）
- **Prism** = 分析层（结构化、洞察、产出）
- **Memory-Core** = 检索层（统一语义搜索，三源整合）
- **导出机制** = 桥梁（定期同步，去重）

**整合方案（推荐 B+C 组合）**：

- **写入流**：Collector 双写 Flomo + Memory-Core（改动最小，新内容立即可分析）
- **检索层**：三源统一导出到 Memory-Core（已部分打通）
- **关键设计**：标签映射、去重机制、时间戳对齐

**战略意义**：

1. **打破数据孤岛**：Flomo（云端）+ Prism（本地）+ Memory-Core（原生）统一检索
2. **自动化知识流**：从"手动收藏"到"自动内化"
3. **可复用的商业模式**："云端知识库 + 自动内容维护"（飞书知识库变现玩法）
4. **Agent 生态基础设施**：知识收集器是 Agent 的"感官"，没有它 AI 就是"闭门造车"

## 实践建议

### 给读者的行动指南

**1. 立即开始收集**

- 安装 `js-knowledge-collector` CLI 工具
- 配置浏览器插件（JS-Eyes），复用登录态
- 遇到好文章，第一时间发送链接入库

**2. 建立三层知识体系**

- **速记层（Flomo）**：快速记录想法、标签检索
- **分析层（Prism）**：结构化分析、产出文章
- **检索层（Memory-Core）**：统一语义搜索，三源整合

**3. 定期维护**

- 每周回顾收集的链接，补充标签
- 每月导出 Flomo 笔记，回灌到本地知识库
- 每季度清理低质量内容，保持知识库精炼

**4. 避免常见坑**

- 即刻/知乎/小红书用 CLI 工具，不要直接用 `knowledge_collect`
- Flomo 推送默认开启，不要每次询问
- Cron 任务处理目录占用和 Git 冲突
