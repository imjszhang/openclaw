## 支撑论点

### 23.1: 大规模上游合并需坚持“架构一致性优先”策略，以高频小步迭代消除冲突并文档化丢失功能

- 逻辑顺序：时间
- 引用 atoms: MU-01, MU-14, MU-15, MV-01, MV-02, MV-17
- 引用 groups: G08

面对横跨五大领域的 3600+ 提交合并，确立以上游版本为准的冲突解决原则，将 shell 等功能损失明确记录为重建路径而非强行保留。通过实施“小步快走”的高频合并策略（4 天间隔），成功将冲突数从 5 个降至 0 个，并建立 Breaking Changes 枚举化与自动迁移机制，确保架构演进平滑可控。

### 23.2: 知识库架构必须严格实行“输入 - 处理 - 输出”三层分离，确保原始素材、结构化拆解与最终产出的职责边界清晰

- 逻辑顺序：结构
- 引用 atoms: KA-06, KA-07, KA-08
- 引用 groups: G03

确立 journal（原始日志不动）、pyramid（结构化拆解）、outputs（面向读者产出）的三层物理隔离架构，防止混合存储导致的边界模糊。特别纠正教程等产出物与拆解过程同级存放的错误，明确其作为 pyramid 视角的输出属性，从而保障知识流转的单向性与纯净度。

## 论点间关系

本 Key Line 遵循“先稳固基石，再构建上层”的结构顺序：首先通过 G08 解决代码库层面的大规模合并与架构对齐问题，为系统提供稳定的运行底座；随后基于 G03 设计清晰的知识库分层架构，确保在稳定底座之上产生的知识资产能够有序流转，共同支撑从“可记录”到“可教学”的战略演进。
