# KL34: 权限配置完全指南

> 所属视角：P23-practice-diary
> 上层论点：必须实施“架构分层分离 + 全链路自动化闭环 + 纵深安全防御”三位一体战略：以五层抽象网关为基石，通过双 Cron 分段驱动与 inbox/batch 轮转实现无人值守的知识产出，同时严格执行执行审批、权限收紧及记忆职责剥离，确保系统从“可记录”平稳演进为“可教学”的智能体。

## 支撑论点

### 34.1: 权限配置需遵循工具可见性与 exec 执行权限的“双轨制”，子代理必须同步 defaults 配置方可正常执行

- 逻辑顺序：结构
- 引用 atoms: PG-01, PG-06, PG-07, PG-08
- 引用 groups: G43

权限系统由工具可见性（allow/deny）与 exec 执行权限（openclaw.json + exec-approvals.json）两个独立维度构成，若 exec-approvals 中 defaults 留空，子代理将 fallback 到 security=deny 导致全部执行被拒。修复方案是在 defaults 中显式设置 security:full 与 ask:off，或使用 agents["*"] 通配符确保双轨配置同步。

### 34.2: 状态目录权限加固需通过 security audit 自动检测与修复，严防配置与凭据被其他用户篡改或读取

- 逻辑顺序：结构
- 引用 atoms: SP-01, SP-02, SP-03, SP-04
- 引用 groups: G44

~/.openclaw 目录存储核心配置与敏感凭据，若对“其他用户”开放读写权限将引发严重的安全风险，需立即执行 `openclaw security audit --deep --fix`。该命令在 Unix 下自动应用 chmod 700/600 策略，在 Windows 下利用 icacls 收紧 ACL，仅保留当前用户与 SYSTEM 权限以消除隐患。

## 论点间关系

两个子论点遵循“运行时逻辑配置”到“静态文件存储安全”的结构顺序：首先解决子代理在执行层面的双轨制权限阻断问题，确保系统“能动”；其次加固底层状态目录的文件系统权限，确保系统“安全”，共同构成完整的权限防御体系。
