# KL49: 第 49 天：JS VI 公众号封面图自动化与品牌一致性

> 所属视角：P23-practice-diary
> 上层论点：必须实施“架构分层分离 + 全链路自动化闭环 + 纵深安全防御”三位一体战略：以五层抽象网关为基石，通过双 Cron 分段驱动与 inbox/batch 轮转实现无人值守的知识产出，同时严格执行执行审批、权限收紧及记忆职责剥离，确保系统从“可记录”平稳演进为“可教学”的智能体。

## 支撑论点

### 49.1: 针对微信平台特殊的像素规范，系统需内建专用尺寸枚举与静态模板约束，以消除人工换算误差并确保品牌视觉统一

- 逻辑顺序：结构
- 引用 atoms: JW-01, JW-02, JW-06, JW-07
- 引用 groups: G71

原有通用海报尺寸缺乏微信专用命名导致运营需自行换算且易损品牌一致性，因此全局 `SIZES` 枚举新增 `wechat-cover` (900×383) 和 `wechat-thumb` (500×500) 键，支持通过 CLI 参数直接切换规格。

### 49.2: 构建独立的 `wechat-cover` 模板目录与结构化元数据，通过禁用动画的静态渲染模式适配直接导出位图的需求

- 逻辑顺序：结构
- 引用 atoms: JW-03, JW-04, JW-05, JW-08, JW-09
- 引用 groups: G71

新增包含 `render.js` 及 `styles.css` 的模板目录，支持 title、tag 等结构化字段填入，并默认禁用动画以确保 `renderToHTML` 能正确内联样式并适配 PNG/JPEG 导出。

### 49.3: 提供 CLI 命令行与 OpenClaw 插件双模调用入口，实现从 Prism 内容定稿到封面生成的全链路自动化闭环

- 逻辑顺序：结构
- 引用 atoms: JW-10, JW-11, JW-12
- 引用 groups: G71

通过独立 CLI 命令及 `vi_poster_generate` 插件支持指定模板与尺寸参数，使得在内容定稿后可直接调用同一套品牌规范生成封面，避免文章与配图割裂。

### 49.4: 明确基于 Puppeteer 的位图导出依赖与 HTML 校对流程，界定视觉系统与后续运营压缩处理的职责边界

- 逻辑顺序：结构
- 引用 atoms: JW-13, JW-14, JW-15
- 引用 groups: G71

导出位图依赖本机 Chrome/Edge 浏览器，而 HTML 格式适合无依赖快速校对，系统仅负责像素与版式一致性，体积压缩策略交由微信编辑器后续处理。

## 论点间关系

本 Key Line 遵循“规范定义 -> 模板实现 -> 调用集成 -> 交付边界”的结构顺序：首先确立微信专用的尺寸与静态规范，继而构建对应的模板工程与元数据支持，随后通过双模接口融入自动化工作流，最后明确导出技术的依赖条件与职责分工，形成完整的自动化闭环。
