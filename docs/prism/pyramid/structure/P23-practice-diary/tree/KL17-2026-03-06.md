# KL35: 第 35 天：知识收集器项目创建

> 所属视角：P23-practice-diary
> 上层论点：必须实施“架构分层分离 + 全链路自动化闭环 + 纵深安全防御”三位一体战略：以五层抽象网关为基石，通过双 Cron 分段驱动与 inbox/batch 轮转实现无人值守的知识产出，同时严格执行执行审批、权限收紧及记忆职责剥离，确保系统从“可记录”平稳演进为“可教学”的智能体。

## 支撑论点

### 35.1: 采用 inbox/batch 轮转与专用 Scraper 架构，彻底解决主会话阻塞并实现多平台全链路自动化

- 逻辑顺序：结构
- 引用 atoms: JK-04, JK-05, JK-16, JK-19, JK-22
- 引用 groups: G52

通过将耗时的抓取与总结任务从主会话剥离至 Cron 隔离处理，利用文件重命名机制实现原子化的批次处理，确保交互流畅；同时针对不同平台（如 Bilibili、X.com）定制专用解析器，完成从 URL 到结构化入库的无人值守闭环。

### 35.2: 构建基于 CLI 显式注册与数据隔离的 Link-Collector 技能，确保并发安全与资源可控

- 逻辑顺序：结构
- 引用 atoms: LC-03, LC-05, LC-07, LC-08, LC-09
- 引用 groups: G45

技能设计遵循“代码只读共享、数据 Workspace 独立”原则，通过用户显式执行 CLI 子命令注册 Cron 任务以控制 Token 消耗，并利用 inbox/batch 文件轮转模型在文件层面实现完全的并发隔离与容错重试。

### 35.3: 改造 js-knowledge-collector 为纯 ESM 插件架构，通过动态路由与按需连接实现 Web UI 无缝集成

- 逻辑顺序：结构
- 引用 atoms: KC-02, KC-05, KC-06, KC-09, KC-10
- 引用 groups: G46

项目重构为标准 OpenClaw 插件结构，摒弃 CommonJS 改用动态 import 解决 ESM 兼容性问题，并通过 `registerHttpRoute` 暴露 Web UI；数据库连接改为按请求开关，配合环境变量注入策略，实现业务代码零改动下的双模式运行。

## 论点间关系

三个子论点遵循“核心业务架构 -> 底层技能实现 -> 上层交互集成”的结构逻辑：首先确立解决阻塞与自动化的核心流水线策略，其次落地保障并发安全与资源控制的技能机制，最后完成面向用户的插件化 Web 集成与兼容性改造。
