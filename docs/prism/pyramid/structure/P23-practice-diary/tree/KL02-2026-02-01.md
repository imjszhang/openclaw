# KL23: 第 2 天：扩展、技能、自定义渠道

> 所属视角：P23-practice-diary
> 上层论点：必须实施“架构分层分离 + 全链路自动化闭环 + 纵深安全防御”三位一体战略：以五层抽象网关为基石，通过双 Cron 分段驱动与 inbox/batch 轮转实现无人值守的知识产出，同时严格执行执行审批、权限收紧及记忆职责剥离，确保系统从“可记录”平稳演进为“可教学”的智能体。

## 支撑论点

### 23.1: 构建基于元数据声明与三级加载机制的动态技能扩展体系

- 逻辑顺序：结构
- 引用 atoms: SG-01, SG-03, SG-13
- 引用 groups: G30

技能系统通过 SKILL.md 前置元数据（如 name, description, openclaw 配置）定义能力边界，并采用“元数据过滤→正文触发→资源按需”的三级加载策略，在保障上下文简洁性的同时实现专业知识的灵活注入。

### 23.2: 实施严格的环境依赖检查与分层优先级覆盖策略

- 逻辑顺序：程度
- 引用 atoms: SG-05, SG-09, SG-10
- 引用 groups: G30

系统依据 bins、env、config 等七项条件进行严格的运行时过滤，并遵循从 Extra 目录到 Plugin 的五级加载优先级（后加载覆盖先加载），确保自定义技能能安全、有序地覆盖或增强内置能力。

### 23.3: 规范技能开发全流程与精细化配置管理

- 逻辑顺序：时间
- 引用 atoms: SG-11, SG-15, SG-18
- 引用 groups: G30

从使用 `skill-creator` 初始化、遵循简洁性最佳实践编写，到自动打包验证及通过 `skills check` 进行故障排查，形成标准化的开发与运维闭环，配合 allowBundled 白名单机制实现最小权限原则下的能力激活。

## 论点间关系

子论点按“核心机制定义（结构）→ 运行控制策略（程度）→ 落地执行流程（时间）”的逻辑展开，阐述了技能系统如何从静态代码库演变为动态、安全且可维护的上下文增强包。
