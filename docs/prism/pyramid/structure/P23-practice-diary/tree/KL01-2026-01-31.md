# KL23: 第 1 天：摸底一个陌生的 AI 框架

> 所属视角：P23-practice-diary
> 上层论点：必须实施“架构分层分离 + 全链路自动化闭环 + 纵深安全防御”三位一体战略：以五层抽象网关为基石，通过双 Cron 分段驱动与 inbox/batch 轮转实现无人值守的知识产出，同时严格执行执行审批、权限收紧及记忆职责剥离，确保系统从“可记录”平稳演进为“可教学”的智能体。

## 支撑论点

### 1.1: 部署基石需遵循“先决条件确认 - 差异化配置 - 状态验证”的标准化流程，以规避多渠道接入的复杂性陷阱

- 逻辑顺序：结构
- 引用 atoms: CD-04, CD-05, CD-08, CD-12, CD-25
- 引用 groups: G12

在配置任何消息渠道前，必须严格确认 Gateway、AI 模型及启动状态三大前置依赖，随后针对 Telegram（Token 型）、WhatsApp（扫码型）及企业协议型渠道的特定陷阱进行差异化配置。最终统一通过 `channels status` 等状态命令完成闭环验证，确保消息路由的可靠性。

### 1.2: 代码演进必须采用"三层分支隔离 + 选择性同步"策略，在平衡上游更新与生产稳定性的同时最小化定制冲突

- 逻辑顺序：结构
- 引用 atoms: FM-03, FM-05, FM-06, FM-22, FM-28
- 引用 groups: G13

通过物理隔离 upstream/main（官方）、origin/main（同步）与 origin/production（生产）三层分支，构建稳定的生产环境底座。采用保守的 cherry-pick 选择性同步策略替代全量合并，并坚持“配置优先、减少源码改动”的分层定制原则，以应对框架 pre-1.0 阶段的频繁迭代。

### 1.3: 智能调度需严格区分"Cron 精确执行”与"Heartbeat 周期感知”的场景边界，构建双驱动自动化任务体系

- 逻辑顺序：结构
- 引用 atoms: CR-03, CR-04, CR-07, CR-18, CR-24
- 引用 groups: G21

利用 Cron 的 `isolated` 会话与 `agentTurn` payload 处理需要精确时间点和完整工具执行的自动化脚本，确保任务独立性与上下文隔离。同时利用 Heartbeat 机制进行低成本的周期性状态感知与批量监控，两者互补形成从日常巡检到精确执行的完整调度闭环。

### 1.4: 自我进化机制需构建"OADA 闭环 + 预算约束 + 人机协作”的三层防御体系，确保 Agent 在主动优化中不失控

- 逻辑顺序：程度
- 引用 atoms: AE-08, AE-15, AE-16, AE-22, AE-24
- 引用 groups: G18

建立从观察、分析、决策到执行、验证（OADA）的全自动进化闭环，使 Agent 从被动响应转向目标导向。在此基础上，严格执行单任务并发控制与 Token/时间双重预算约束，并保留人类在关键决策点的干预权与一键回滚能力，防止自动化迭代导致资源耗尽或逻辑偏离。

### 1.5: 安全防御需落实“本地中继隔离 + 权限最小化 + 全栈可观测”策略，筑牢从浏览器控制到资源消耗的防线

- 逻辑顺序：结构
- 引用 atoms: BR-03, BR-19, OD-16, TM-06, TM-10
- 引用 groups: G20, G19, G16

通过 Browser Relay 的本地中继服务器与扩展徽章状态机，实现 AI 对浏览器标签页的安全可控共享，杜绝直接暴露 CDP 端口。结合 Doctor 工具的深度安全审计与 OpenTelemetry 的全栈指标导出，实时监控权限配置、网络暴露风险及 Token 消耗趋势，确保系统在透明可控的状态下运行。

## 论点间关系

本 Key Line 按照“基础部署标准化 → 代码架构稳定性 → 核心调度自动化 → 高级进化可控性 → 全域安全可观测”的结构逻辑展开。从底层的渠道接入与代码管理入手，逐步构建自动化的任务调度与自我进化能力，最后以全方位的安全防御与监控体系作为保障，形成完整的智能体落地闭环。
