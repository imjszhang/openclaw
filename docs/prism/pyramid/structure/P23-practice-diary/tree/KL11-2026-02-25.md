# KL26: 第 26 天：爆发，五件事同时发生

> 所属视角：P23-practice-diary
> 上层论点：必须实施“架构分层分离 + 全链路自动化闭环 + 纵深安全防御”三位一体战略：以五层抽象网关为基石，通过双 Cron 分段驱动与 inbox/batch 轮转实现无人值守的知识产出，同时严格执行执行审批、权限收紧及记忆职责剥离，确保系统从“可记录”平稳演进为“可教学”的智能体。

## 支撑论点

### 26.1: OpenClaw 插件创建需遵循"SDK 隔离 + 清单驱动 + 安全加载”的全生命周期规范

- 逻辑顺序：结构
- 引用 atoms: PC-01, PC-03, PC-04, PC-07, PC-30
- 引用 groups: G32

插件系统通过严格的 SDK 隔离（禁止导入内部模块）与 jiti 动态加载机制，确立了编译时稳定与运行时注入的双层架构。全生命周期包含从清单验证、ID 冲突检查到世界可写路径拒绝的十步安全流程，确保扩展能力不破坏系统稳定性。

### 26.2: 知识棱镜的自动化落地需通过"OpenClaw 插件化”将繁琐的手动流程转化为 AI 可调用的标准工具

- 逻辑顺序：时间
- 引用 atoms: FN-06, FN-10, FN-16, UP-04, UP-19
- 引用 groups: G33

针对手动处理 journal 素材的繁琐痛点，将知识棱镜流程封装为 `knowledge_prism_process` 等标准 AI 工具，消除了终端切换的上下文成本。该插件支持分阶段（Atoms/Groups/Synthesis）执行与幂等处理，并建立了 git commit 前审查 diff 的自动化维护工作流。

### 26.3: JS-Eyes 通过 WebSocket 复用用户浏览器环境，填补了内置沙箱浏览器在“带登录态交互”场景的能力空白

- 逻辑顺序：结构
- 引用 atoms: JE-01, JE-03, JE-26, JE-19, JE-24
- 引用 groups: G34

JS-Eyes 利用 WebSocket 复用用户日常浏览器（Chrome/Edge/Firefox）的现有会话状态，解决了内置 CDP 浏览器无法继承登录态的局限。通过封装 7 个核心工具（如获取 Cookie、注入脚本），实现了 Agent 对复杂网页的无损控制，并提供了端口占用排查等故障诊断机制。

### 26.4: 技能分发体系需构建“自主安装脚本 + 多源回退链 + 静态注册表”的独立架构以解耦市场依赖并实现 Agent 友好

- 逻辑顺序：程度
- 引用 atoms: AF-07, AF-13, AF-20, SD-04, SD-12
- 引用 groups: G36, G37

为规避 ClawHub 市场的安全误报与网络限制，构建了基于 curl/PowerShell 的自主安装脚本与“自有域名→GitHub→jsDelivr"的多源回退链。同时，演进出的 GitHub Pages 静态 `skills.json` 注册表作为唯一真相来源，配合 AI 发现工具，实现了从手动配置到 Agent 自主感知的架构升级。

### 26.5: OpenClaw 路径解析存在 State 目录与 Workspace 目录的分离机制，需通过环境变量或配置文件显式对齐以避免 ENOENT 错误

- 逻辑顺序：结构
- 引用 atoms: OP-01, OP-02, OP-04, OP-05, OP-07
- 引用 groups: G38

系统默认将 State 目录（由 `OPENCLAW_STATE_DIR` 控制）与 Workspace 目录（由 `OPENCLAW_HOME` 或系统家目录控制）分离解析，导致跨盘符部署时出现 ENOENT 错误。必须通过设置 `OPENCLAW_HOME` 环境变量或在 `openclaw.json` 中显式指定 `agents.defaults.workspace` 来强制对齐路径。

## 论点间关系

本 Key Line 遵循“基础规范 -> 核心应用 -> 能力扩展 -> 分发架构 -> 底层修复”的结构逻辑顺序。首先确立插件开发的安全基石（26.1），随即落地核心业务自动化（26.2）并扩展浏览器交互能力（26.3）；进而构建独立于第三方市场的分发与发现体系（26.4），最后修复阻碍系统运行的底层路径解析缺陷（26.5），共同支撑系统向“可教学”智能体的演进。
