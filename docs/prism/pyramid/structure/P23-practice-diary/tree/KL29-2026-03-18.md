# KL47: Cursor Agent 集成必须采用“零依赖核心层 + 长驻进程池 + 多入口适配”架构以解决短命令模型与有状态会话的冲突

> 所属视角：P23-practice-diary
> 上层论点：必须实施“架构分层分离 + 全链路自动化闭环 + 纵深安全防御”三位一体战略：以五层抽象网关为基石，通过双 Cron 分段驱动与 inbox/batch 轮转实现无人值守的知识产出，同时严格执行执行审批、权限收紧及记忆职责剥离，确保系统从“可记录”平稳演进为“可教学”的智能体。

## 支撑论点

### 1.1: 明确短命令模型与 Cursor 长驻有状态进程的根本冲突，确立独立进程管理方案的必要性

- 逻辑顺序：结构
- 引用 atoms: JA-03, JA-05
- 引用 groups: G68

现有 acpx 插件的"spawn-exec-exit"短命令模式无法兼容 Cursor agent 基于 stdio JSON-RPC 的长驻有状态会话模型，直接在配置中修补或添加条目均不可行，必须构建独立的长驻进程管理方案以支撑持续的会话交互。

### 1.2: 构建“零依赖核心层 + 多入口适配”架构，实现业务逻辑与具体场景的彻底解耦

- 逻辑顺序：结构
- 引用 atoms: JA-06, JA-07, JA-19
- 引用 groups: G68

采用“一套核心逻辑 + 三种消费入口（CLI/MCP/Plugin）”架构，将核心层设计为零 OpenClaw 依赖的纯 JavaScript 模块，仅由插件层作为薄适配器桥接，确保所有智能集中在 core 层而避免项目绑死特定平台。

### 1.3: 实现具备自动回收与权限策略的长驻进程池，支撑高并发下的稳定会话复用

- 逻辑顺序：程度
- 引用 atoms: JA-08, JA-12, JA-17, JA-11
- 引用 groups: G68

核心层通过 ProcessManager 实现按会话映射的进程池，设置默认 4 个并发限制并驱逐最老进程，每分钟自动回收空闲超 30 分钟的实例，同时内置 approve-all/read/deny 三档权限策略以在非交互环境下自动审批。

### 1.4: 自研轻量级 JSON-RPC 传输层与流式客户端封装，适配 Cursor 特有协议与交互模式

- 逻辑顺序：结构
- 引用 atoms: JA-01, JA-10, JA-16, JA-13, JA-18
- 引用 groups: G68

摒弃官方 SDK 依赖，利用 readline 自实现 JSON-RPC 2.0 传输层以处理双向通信及无 ID 通知，并封装支持 AsyncGenerator 的 prompt 方法，在等待响应期间流式 yield text_delta 与 tool_call 事件以适配 Cursor 的实时交互需求。

### 1.5: 提供 MCP 双模式入口与 OpenClaw 运行时桥接，完成多场景复用与技术选型落地

- 逻辑顺序：结构
- 引用 atoms: JA-14, JA-15, JA-09, JA-20
- 引用 groups: G68

MCP Server 提供 stdio/HTTP 双模式入口包含 6 个管理工具，OpenClaw 插件通过实现 AcpRuntime 接口桥接核心层，选用纯 JavaScript 保持风格一致，并预留会话持久化绑定 channel/thread 的演化路径以实现跨重启恢复。

## 论点间关系

本 Key Line 遵循“问题定义与架构决策 -> 核心层实现细节 -> 多入口适配方案 -> 运维与演化策略”的结构顺序：首先界定短命令与长驻模型的冲突并确立分层架构，接着详述进程池管理与传输层实现等核心机制，随后展开 MCP 与插件的具体适配方式，最后补充技术选型理由及未来持久化方向。
