# KL27: 踩坑密集日

> 所属视角：P23-practice-diary
> 上层论点：必须实施“架构分层分离 + 全链路自动化闭环 + 纵深安全防御”三位一体战略：以五层抽象网关为基石，通过双 Cron 分段驱动与 inbox/batch 轮转实现无人值守的知识产出，同时严格执行执行审批、权限收紧及记忆职责剥离，确保系统从“可记录”平稳演进为“可教学”的智能体。

## 支撑论点

### 27.1: OpenClaw 的 exec 安全机制通过“分段解析 + 白名单校验 + 动态审批”三重防线，强制阻断高风险的 curl|bash 管道执行模式

- 逻辑顺序：结构
- 引用 atoms: EX-01, EX-02, EX-03, EX-04, EX-16
- 引用 groups: G39

系统默认将含管道符的命令拆解为独立段进行严格校验，由于 bash 和 curl 不在默认 safeBins 且具备高风险特性，除非显式配置 allowlist 或启用审批，否则直接阻断。这种设计哲学故意不自动放行高危管道模式，迫使开发者必须明确知晓并授权每一个执行步骤。

### 27.2: 执行审批策略需显式配置"allowlist+on-miss"组合，避免因空配置回退导致的静默拒绝陷阱

- 逻辑顺序：程度
- 引用 atoms: EX-05, EX-06, EX-07, EX-15
- 引用 groups: G39

在 exec-approvals.json 中，若 agents 配置为空或未显式指定 agent，系统不会采用宽松策略，而是回退到 security=deny 直接拒绝所有请求。必须为 main agent 显式设置 security 为 allowlist 并将 ask 设为 on-miss，才能在白名单未命中时弹出审批窗口而非直接失败。

### 27.3: 运行时环境依赖与配置持久化机制存在多重隐蔽陷阱，需严格校验 host 设置并重启 Session 以生效

- 逻辑顺序：时间
- 引用 atoms: EX-08, EX-09, EX-10, EX-11, EX-12, EX-13, EX-14
- 引用 groups: G39

审批逻辑仅在 tools.exec.host 显式设置为 gateway 或 node 时生效，默认 sandbox 模式会绕过所有检查；且配置修改后必须开启新 Session 才能清除旧状态中的持久化覆盖值。此外，首次被拒命令可通过审批弹窗的"Always allow"自动加入白名单，或手动追加路径模式对象实现永久放行。

## 论点间关系

本 Key Line 遵循“安全原理机制 -> 配置策略陷阱 -> 运行时环境依赖”的结构顺序展开：首先阐述系统如何通过分段解析和默认白名单从原理上阻断高危命令，接着揭示空配置回退和审批模式选择的策略性陷阱，最后说明 host 环境设置与 Session 状态持久化对配置生效的关键影响。
