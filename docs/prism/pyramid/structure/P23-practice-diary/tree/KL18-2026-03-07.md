# KL36: 多知识库注册与 Cron 自动化

> 所属视角：P23-practice-diary
> 上层论点：必须实施“架构分层分离 + 全链路自动化闭环 + 纵深安全防御”三位一体战略：以五层抽象网关为基石，通过双 Cron 分段驱动与 inbox/batch 轮转实现无人值守的知识产出，同时严格执行执行审批、权限收紧及记忆职责剥离，确保系统从“可记录”平稳演进为“可教学”的智能体。

## 支撑论点

### 36.1: 突破单库限制，构建基于注册表与 Cron 的批量自动化运维体系

- 逻辑顺序：结构
- 引用 atoms: RC-01, RC-02, RC-07, RC-08
- 引用 groups: G50

针对原有插件仅支持单一 `baseDir` 且缺乏定时调度能力的痛点，参考 `js-knowledge-collector` 的三层协作模式，新增 4 个 CLI 子命令与 AI 工具，并定义 `prism-processor` 技能，实现了从手动单库操作向多库自动批量处理的架构升级。

### 36.2: 坚持核心层职责分离，利用原子操作与幂等机制保障执行安全

- 逻辑顺序：程度
- 引用 atoms: RC-03, RC-04, RC-09, RC-12
- 引用 groups: G50

严格界定插件层（Registry）与核心层（lib/）的边界，核心库始终面向单目录；通过 `tmp + rename` 原子写入注册表、复用 `runPipeline` 天然幂等性以及避免硬编码路径，确保在多库并发场景下的数据一致性与系统鲁棒性。

### 36.3: 优化部署与交互流程，以低开销策略实现无人值守的知识产出闭环

- 逻辑顺序：时间
- 引用 atoms: RC-05, RC-06, RC-10, RC-11
- 引用 groups: G50

采用 `process_all` 封装批量操作以减少 LLM 往返开销，结合多级路径 fallback 解析与“重启网关→注册→配置”的标准部署流，使代码改动通过路径链接直接生效，最终达成无需人工干预的自动化知识采集目标。

## 论点间关系

本 Key Line 遵循“问题重构→架构防御→流程落地”的结构顺序：首先通过引入注册表和 Cron 解决单库瓶颈（36.1），继而通过职责分离和原子操作确立安全基石（36.2），最后通过优化交互与部署流程完成自动化闭环的落地验证（36.3）。
