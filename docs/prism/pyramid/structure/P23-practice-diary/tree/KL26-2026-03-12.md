# KL41: 模板创建技能决策引导与渐进式披露策略

> 所属视角：P23-practice-diary
> 上层论点：必须实施“架构分层分离 + 全链路自动化闭环 + 纵深安全防御”三位一体战略：以五层抽象网关为基石，通过双 Cron 分段驱动与 inbox/batch 轮转实现无人值守的知识产出，同时严格执行执行审批、权限收紧及记忆职责剥离，确保系统从“可记录”平稳演进为“可教学”的智能体。

## 支撑论点

### 41.1: 针对多层知识门槛，应设计专用技能通过“四问映射”引导配置，而非仅提供 CLI 工具

- 逻辑顺序：结构
- 引用 atoms: PT-01, PT-02
- 引用 groups: G61

模板创建涉及产出形态、写作风格等多层知识，单纯 CLI 工具无法解决新用户的认知负荷。专用技能的核心价值在于将创意任务（Prompt 编写）留给 LLM，而将结构化的配置决策通过引导流程固化下来。

### 41.2: 技能架构需采用“决策引导 +Schema 参考+Scaffold 工具”三层分离设计

- 逻辑顺序：结构
- 引用 atoms: PT-03, PT-04
- 引用 groups: G61

通过 SKILL.md 承载决策引导逻辑，利用四个关键问题（产出形态、风格、复杂度、审校）直接映射到具体的 split、type 和组件配置。Schema 参考作为附件存在，确保主流程清晰且配置准确。

### 41.3: 核心 Scaffold 工具必须执行纯确定性逻辑，严格剥离内容生成职责

- 逻辑顺序：程度
- 引用 atoms: PT-05, PT-06, PT-08
- 引用 groups: G61

`prism_scaffold_template` 和 `prism_scaffold_component` 仅负责校验命名、合并默认值、生成 frontmatter 及占位文件，绝不调用 LLM。这种确定性逻辑保证了文件结构的绝对正确，将内容填充完全交由 Agent 根据引导完成。

### 41.4: 实施渐进式披露策略，控制主文件行数并按需加载详细 Schema

- 逻辑顺序：程度
- 引用 atoms: PT-07, PT-12
- 引用 groups: G61

将主技能文件（SKILL.md）严格控制在 130 行以内，避免信息过载。详细的 schema 定义放置在 `schema-reference.md` 等附件中，仅在 Agent 需要深入配置时按需查阅，保持交互界面的简洁性。

### 41.5: 规范文件写入路径与参数接口，并建立 Dry-run 验证机制

- 逻辑顺序：时间
- 引用 atoms: PT-09, PT-10, PT-11
- 引用 groups: G61

工具优先将模板写入知识库 `outputs/_templates/` 目录，缺失时回退至项目内置目录。执行时需传入 name、split、type 等标准参数，并强制通过 `--dry-run` 参数验证生成结果的正确性后方可正式落地。

## 论点间关系

本 Key Line 遵循“问题定义 -> 架构设计 -> 核心逻辑 -> 交互策略 -> 落地规范”的结构顺序。首先明确高门槛痛点与 LLM 的分工边界，进而提出三层架构方案；接着强调工具必须严守确定性逻辑以保障结构安全，配合渐进式披露优化体验；最后落实到具体的路径规范与验证闭环，确保技能可执行、可验证。
