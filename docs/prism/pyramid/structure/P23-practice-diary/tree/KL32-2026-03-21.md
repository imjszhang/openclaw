# KL50: 上游 PR 分支隔离与插件架构解耦

> 所属视角：P23-practice-diary
> 上层论点：必须实施“架构分层分离 + 全链路自动化闭环 + 纵深安全防御”三位一体战略：以五层抽象网关为基石，通过双 Cron 分段驱动与 inbox/batch 轮转实现无人值守的知识产出，同时严格执行执行审批、权限收紧及记忆职责剥离，确保系统从“可记录”平稳演进为“可教学”的智能体。

## 支撑论点

### 50.1: 上游 PR 贡献必须采用「独立分支隔离 + Cherry-pick 回合并」策略以规避定制线冲突

- 逻辑顺序：时间
- 引用 atoms: OA-01, OA-02, OA-06, OA-07, OA-08, OA-09, OA-10, OA-11, OA-12, OA-13
- 引用 groups: G72

向官方上游提交 PR 时，需基于 `source/main` 创建独立分支以避免与本 fork 定制线混淆，并在合回本地长期分支时严禁直接 Merge，改用 Cherry-pick 提取特定提交以消除无关差异带来的高冲突成本。该流程涵盖了从分支创建、推送追踪到冲突中止与选择性合并的全生命周期操作，显著降低了维护分叉版本的复杂度。

### 50.2: 插件注册表架构必须严格分离「类型契约声明」与「运行时装配逻辑」并实施细粒度权限控制

- 逻辑顺序：结构
- 引用 atoms: OT-01, OT-02, OT-03, OT-04, OT-05, OT-06, OT-07, OT-08, OT-09, OT-10, OT-11, OT-12, OT-13, OT-14, OT-15, OT-16
- 引用 groups: G73

通过将 `types.ts`（声明契约）与 `registry.ts`（执行装配）物理拆分，明确了模块依赖边界，并利用注册模式枚举、路由冲突检测及提示注入过滤机制构建了安全可控的运行时环境。这种架构设计不仅防止了误触全局副作用和路由抢占，还通过强制剥离危险字段确保了插件系统在复杂交互下的数据完整性与安全性。

## 论点间关系

两个子论点按“外部协作规范”到“内部架构治理”的结构顺序组织：前者解决了开源协作中代码合并的冲突痛点，后者确立了系统内部插件运行的安全基石，共同支撑起系统从单一记录向可教学智能体演进所需的稳定性与安全性。
