# 当 666 权限拦住了我的插件：独立分发体系的惊险一跃

> Day 29 · 2026-02-28 · 骨架 [KL14](../../../pyramid/structure/P23-practice-diary/tree/KL14-2026-02-28.md)

昨天刚把 js-eyes 的分发体系跑通，今天本想趁热打铁探索 Agent 的长期记忆，结果一上来就被一个看似不起眼的权限问题给“教做人”了。独立分发脚本的健壮性，果然比预想中要苛刻得多。

## 666 权限引发的“插件失踪”案

早上测试子技能 `js-search-x` 的安装流程时，脚本明明提示成功，配置也按要求写进了 `~/.openclaw/openclaw.json`，但执行 `openclaw daemon restart` 时却直接崩了。报错信息非常诡异：`plugin not found: js-search-x`。明明文件就在那儿，怎么系统就说找不到？

跑了一把 `openclaw doctor` 才真相大白：`blocked plugin candidate: world-writable path`。原来是我打包 `js-search-x-skill.zip` 时手松了，里面的 `index.mjs` 等核心文件权限全是 `0666`（所有人可写）。OpenClaw 的安全机制 `checkPathStatAndPermissions` 极其严格，一旦发现插件路径下有任何文件是 "other-writable"，直接拒绝加载，连带着配置校验里的插件 ID 也解析失败，导致整个守护进程启动流产。

这算是给我提了个醒：在自动化分发体系里，解压后的权限清洗是绝对不能省的一步。我在 `install.sh` 的解压逻辑后紧急补了两行 `find` 命令，强制将文件收紧为 `644`，目录设为 `755`：

```bash
find "$TARGET" -type f -exec chmod 644 {} + 2>/dev/null || true
find "$TARGET" -type d -exec chmod 755 {} + 2>/dev/null || true
```

加上这两行后，重启瞬间成功。看来以后每个解压分支都得加上这道“安检门”，不然用户环境千奇百怪，指不定哪步就踩雷。

## 给记忆系统立规矩：从“自动写”到“治理闭环”

解决了拦路虎，下午终于能安心研究 `memory-core` 了。之前一直有个误区，以为开了记忆插件就能自动把什么都记下来，结果调研发现 `memory-core` 本质上只是个“索引同步层”，它不负责语义化的存储，更不会自动把会话内容变成记忆条目。

如果不加干预，指望它自动工作，最后只能得到一堆混乱的索引。于是我决定给记忆体系立个规矩：**写入和检索必须解耦，且要有明确的治理周期**。

我在 `~/.openclaw` 运行侧做了一番改造：

1. **定模板**：新增 `memory/_TEMPLATE.md` 和 `MEMORY.md` 规范，明确什么值得记，什么只是日志。
2. **建管道**：写了个 `scripts/memory_digest.py`，每天定时从执行日志里抽取高价值事件，生成正式的 `memory/YYYY-MM-DD.md`，而不是让 Agent 胡乱写。
3. **定策略**：在 `HEARTBEAT.md` 里明确“先搜后答”的原则，并新增 `memory/RETRIEVAL.md` 固化高频查询模板。
4. **周复盘**：搞了个 `memory_weekly_review.py`，每周自动跑一次，统计命中率、噪声率和可行动率，逼着自己优化记忆质量。

这一套组合拳下来，记忆系统才算从“能用”变成了“可维护”。毕竟，长期记忆的价值不在于存了多少，而在于能不能在需要的时候精准捞出来。

## 今天的收获

- 独立分发脚本必须包含解压后的权限清洗步骤，否则会被宿主系统的安全机制直接拦截。
- OpenClaw 的插件加载器对 `world-writable` 零容忍，这是防止恶意篡改的底线而非可选项。
- 长期记忆系统的核心不是“自动记录”，而是“高质量输入源”与“定期治理”的闭环。
- 将心跳检测（Heartbeat）与记忆写入解耦，能避免状态日志污染长期知识库。
- 没有量化指标（如命中率、噪声率）的记忆系统，最终都会退化为不可用的数据垃圾场。
