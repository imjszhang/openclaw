# 把模板创作从“配置地狱”变成“四问引导”

> Day 41 · 2026-03-12 · 骨架 [KL26](../../../pyramid/structure/P23-practice-diary/tree/KL26-2026-03-12.md)

昨天刚把 Draft 机制的时区坑填平，今天本来想顺势多跑几个输出，结果在准备新模板时卡住了——创建一个新模板竟然要同时搞定 frontmatter、区段变量、组件引用和审校逻辑，这门槛高得离谱。与其让用户对着文档猜配置，不如直接让 LLM 来当这个“向导”，于是今天的主线任务变成了：设计一个能自动决策的模板创建技能。

## 为什么 CLI 救不了模板创作，但技能可以

一开始我习惯性地想加几个 CLI 参数，比如 `--type blog --style narrative`，但马上意识到这只是在转移痛苦。用户还是得知道 `blog` 对应什么 split，`narrative` 需要哪个 persona 组件。LLM 真正的强项不是生成空骨架，而是理解“我想写一篇像技术周记那样的文章”这种模糊意图，并把它翻译成具体的配置。

所以我决定放弃纯工具思路，转而开发 `prism-template-author` 技能。核心逻辑变了：不再是用户填参数，而是技能反问用户四个问题——产出形态（是单篇还是系列？）、写作风格（叙事还是教程？）、生成复杂度（一步到位还是分阶段？）、是否需要审校？这四个问题的答案直接映射到内部的 split、type 和组件配置。用户只需要关注业务意图，剩下的技术细节全被封装在技能的决策树里了。

## 确定性 Scaffold 与 Agent 创意的“楚河汉界”

技能设计好了，具体落地时又面临一个架构抉择：文件结构谁来生成？Prompt 内容谁来写？如果全交给 LLM，万一它少写个 frontmatter 字段或者区段标题拼错，整个模板就废了；如果全写死，又失去了灵活性。

最后的方案是搞“严格分工”。我写了两个纯确定性的 AI 工具：`prism_scaffold_template` 和 `prism_scaffold_component`。这两个工具**绝不调用 LLM**，只负责校验模板名是否重复、合并默认值、生成标准的 frontmatter 和区段占位符。它们就像建筑工地的钢筋工，保证骨架绝对横平竖直。而具体的 Persona 设定、Style 描述这些需要“创意”的内容，则完全交给 Agent 根据技能引导去填充。这种“确定性构建骨架 + Agent 负责内容”的策略，既保证了文件的结构正确性，又释放了 LLM 在创意生成上的价值。

为了不让技能文件本身变得臃肿，我还实施了渐进式披露：主技能文件 `SKILL.md` 控制在 130 行以内，只放决策流程；详细的 Schema 参考、变量表全部扔到附件 `schema-reference.md` 里，Agent 只有在需要查表时才会去读。最后生成完模板，再用 `npx js-knowledge-prism output --dry-run` 跑一遍预验证，确保落盘前逻辑无误。

## 今天的收获

- 模板创建的高门槛不能靠堆砌 CLI 参数解决，必须用专用技能进行意图到配置的自动映射。
- LLM 的核心价值在于生成高质量的创意内容（如 Prompt），而非生成容易出错的结构骨架。
- 核心工具链必须严格遵循“纯确定性逻辑构建骨架，Agent 负责内容填充”的分工策略。
- 采用渐进式披露策略，将主技能轻量化，详细 Schema 作为附件按需查阅，可显著降低认知负荷。
- 在模板正式落盘前，强制通过 dry-run 机制进行预验证是确保结构正确性的关键质检点。
