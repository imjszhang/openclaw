# 把创意留给 LLM，把结构交给代码：模板创建技能的诞生

> Day 41 · 2026-03-12 · 骨架 [KL26](../../../pyramid/structure/P23-practice-diary/tree/KL26-2026-03-12.md)

昨天还在纠结怎么让新用户理解 frontmatter 和区段结构，今天决定不再写枯燥的文档，而是直接打造一个能“对话”的专用技能。既然模板创建涉及产出形态、写作风格等多层认知负荷，不如让 Agent 来引导决策，我们只负责把路铺平。

## 为什么我拒绝只做一个 CLI 脚手架

一开始的想法很简单：写个命令行工具，用户输入 `create-template --name demo`，生成一堆空文件完事。但马上我就意识到这行不通。Prompt 编写本质上是个创意任务，LLM 最有价值的地方是生成高质量的内容，而不是仅仅吐出一个冷冰冰的空骨架。如果用户连 split 选 `per-kl` 还是 `per-perspective` 都搞不清楚，给再完美的目录结构也是白搭。

所以我砍掉了纯 CLI 的方案，转而设计了一个三层架构的 `prism-template-author` 技能。核心思路是“分工”：用 SKILL.md 承载决策逻辑，通过四个关键问题（产出形态、写作风格、生成复杂度、质量审校）把用户的模糊意图映射到具体的配置上；详细的 schema 定义扔进 `schema-reference.md` 附件里，Agent 按需查阅；而真正的文件生成，则交给两个纯确定性的 Scaffold 工具。这样一来，创意部分完全由 Agent 根据引导完成，而结构正确性则由代码死死守住。

## 让 Scaffold 工具“闭嘴”，只做确定性执行

在实现 `prism_scaffold_template` 和 `prism_scaffold_component` 这两个工具时，我给自己立了一条铁律：绝不调用 LLM。

这听起来有点反直觉，毕竟这是个 AI 项目。但我发现，一旦让 LLM 参与文件结构的生成，它就会开始“幻觉”，比如编造不存在的 type，或者把 frontmatter 的字段顺序搞乱。于是我把这两个工具做成了纯粹的“执行者”：它们只负责校验模板名是否重复、合并 type 的默认值、生成标准的 frontmatter 和区段占位符。

比如执行 `prism_scaffold_template({ name: "tutorial", split: "per-perspective", type: "blog" })` 时，工具不会思考“tutorial"该写什么内容，它只会机械地检查 `outputs/_templates/` 目录下有没有重名，然后精准地吐出文件路径。这种“笨拙”的确定性逻辑，反而保证了系统从“可记录”向“可教学”演进时的底线安全。内容填充？那是 Agent 拿着技能引导去发挥的事，脚手架绝不越俎代庖。

## 今天的收获

- 高门槛的创意任务不适合纯 CLI 工具，应设计专用技能通过“四问映射”引导用户决策。
- 技能架构需将决策引导、Schema 参考与 Scaffold 工具三层分离，保持主流程清晰。
- 核心 Scaffold 工具必须执行纯确定性逻辑，严格剥离内容生成职责以保障结构安全。
- 实施渐进式披露策略，将主技能文件控制在 130 行以内，详细 Schema 按需加载。
- 建立“本地优先”的写入路径规范与 `--dry-run` 验证机制，确保生成结果可预期。
