# 破局短命令模型：Cursor Agent 零依赖架构与长驻进程池实战

> Day 47 · 2026-03-18 · 骨架 [KL29](./P23-practice-diary/KL29-2026-03-18.md)

今天的核心任务是解决 OpenClaw 驱动 Cursor IDE 的架构冲突：现有的 `acpx` 插件采用"spawn-exec-exit"短命令模式，而 Cursor 的 `agent acp` 基于 stdio JSON-RPC 2.0 需要长驻有状态会话。我放弃了在配置字典中强行修补的捷径，转而构建了独立的 `js-cursor-agent` 项目，通过“零依赖核心层 + 长驻进程池”架构彻底解耦业务逻辑与运行环境。

## 短命令模型与有状态会话的根本冲突

在分析 OpenClaw 的 ACP 系统时，我确认了需求属于"ACP Runtime"模式，即让 OpenClaw 作为客户端驱动外部工具。然而，现有的 `acpx` 插件是为 Codex、Claude 等 harness agents 设计的，其生命周期是每次操作 spawn 一个新进程，执行完立即退出。这与 Cursor agent 的模型存在根本性冲突：Cursor 启动后需保持运行，在同一进程内创建多个 session 并切换模式，且运行时会动态发起 `session/request_permission` 权限请求。试图直接在 `ACPX_BUILTIN_AGENT_COMMANDS` 中添加 Cursor 条目不仅不可行，还会破坏原有的稳定性。这一认知迫使我确立独立进程管理方案的必要性，必须构建一个能维持长连接并处理双向通信的专用层。

## 构建“零依赖核心层 + 多入口适配”架构

为了避免项目被绑死在 OpenClaw 生态中，我参考了 `js-knowledge-flomo` 的模式，设计了“一套核心逻辑 + 三种消费入口”的架构。核心层 `core/` 被严格设计为纯 JavaScript 模块，零 OpenClaw 依赖，仅包含 `config.js`、`jsonrpc.js`、`process-manager.js` 等基础逻辑。在此之上，我构建了三个薄适配器入口：独立的 CLI 工具、MCP Server 以及 OpenClaw 插件。这种分层确保了所有智能集中在 core 层，插件层（如 `openclaw-plugin/index.mjs` 中的 `CursorRuntime` 类）仅负责实现 `AcpRuntime` 接口进行桥接，不包含任何业务逻辑，从而实现了真正的技术栈解耦。

## 长驻进程池的自动回收与权限策略

核心层的 `process-manager.js` 是实现高并发稳定性的关键。与 acpx 的一次性进程不同，我实现了一个按 sessionKey 映射的长驻进程池。该管理器支持 `getOrSpawn` 复用已有进程，并设置了默认 4 个并发限制，超出时自动驱逐最老的进程。为了防止资源泄漏，我加入了每分钟扫描机制，自动回收空闲超过 30 分钟的实例。针对非交互环境下的权限阻塞问题，我在 spawn 时自动注入了权限处理器，内置了 `approve-all`、`approve-reads` 和 `deny-all` 三档策略，确保在无人值守的自动化流程中，Cursor 的工具调用请求能被即时审批或拒绝，避免会话挂起。

## 自研轻量级 JSON-RPC 传输层与流式封装

在传输层实现上，我摒弃了厚重的官方 SDK，利用 Node.js 原生的 `readline` 和 `JSON.parse` 自实现了 JSON-RPC 2.0 传输层。这个 `JsonRpcTransport` 模块不仅要处理带 ID 的出站请求（通过自增 ID 和 pending Map 管理超时），还要精准分发入站响应和无 ID 通知（如 `session/update` 流式事件）。基于此，我在 `acp-client.js` 中将 `prompt` 方法封装为 `AsyncGenerator`。在等待 Cursor 响应期间，它能从内部队列中实时 yield 出 `text_delta` 和 `tool_call` 事件，完美适配了 Cursor 的实时交互需求，同时支持通过 `AbortSignal` 进行取消操作。

## MCP 双模式入口与 OpenClaw 运行时桥接

为了完成多场景落地，我实现了 MCP Server 的 stdio 和 HTTP 双模式入口，提供了包括 `cursor_session_new`、`cursor_prompt` 在内的 6 个管理工具，使得 Claude Desktop 等其他 IDE 也能调用 Cursor 能力。在 OpenClaw 插件侧，`CursorRuntime` 类严格实现了 `ensureSession`、`runTurn`、`cancel` 等标准接口，将核心层的异步生成器事件翻译为 `AcpRuntimeEvent` 格式。目前项目已包含 24 个文件，从核心逻辑到 Web 状态面板（`src/index.html`）全部采用纯 JavaScript 编写，保持了与团队现有项目风格的一致性，并为未来将会话持久化绑定到 OpenClaw channel/thread 预留了演化路径。

## 今天的收获

- **架构分层是解耦的关键**：将核心逻辑剥离为零依赖模块（core/），通过薄适配器支撑 CLI/MCP/Plugin 多入口，避免了特定平台的绑定风险。
- **进程模型决定集成方案**：面对长驻有状态服务（如 Cursor agent），必须放弃短命令插件模式，转而构建具备自动回收机制的独立进程池。
- **传输层需按需自研**：对于特定的 JSON-RPC 流式交互，自研轻量级传输层（readline + JSON.parse）比引入重型 SDK 更灵活且易于控制超时与通知。
- **非交互环境的权限治理**：在自动化场景中，必须内置多档权限审批策略（approve-all/read/deny）以防止会话因等待用户确认而阻塞。
- **流式输出的 AsyncGenerator 模式**：将长轮询或长连接操作封装为 AsyncGenerator，是处理实时文本增量和工具调用事件的最佳实践。
