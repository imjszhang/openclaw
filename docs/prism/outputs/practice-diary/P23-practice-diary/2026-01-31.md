# 摸底一个陌生 AI 框架：从架构报告到 TinyCore 极限部署

> Day 1 · 2026-01-31 · 骨架 [KL01](../../../pyramid/structure/P23-practice-diary/tree/KL01-2026-01-31.md)

今天是 OpenClaw 实践日记的第一天，目标很明确：在不看官方文档的前提下，通过阅读项目源码和分析报告，彻底摸清这个“本地优先”AI 助手平台的底细，并评估它在极端资源受限环境下的生存能力。

## 架构分析报告：理清“本地优先”的全链路数据流

上手一个陌生框架，我最怕直接扎进代码细节里迷路。所以第一步，我强迫自己先读完 `openclaw-analysis-report.md`。这份报告像一张高精度的地图，瞬间帮我建立了全局认知。

报告里最核心的概念是“本地优先 Gateway"。以前我理解的 AI 助手大多是云端 API 调用，但 OpenClaw 把控制平面完全本地化。数据流非常清晰：Channels（消息渠道，如 WhatsApp、Telegram）作为入口，将消息汇聚到本地的 Gateway（默认监听 18789 端口），Gateway 再根据路由规则分发给 Agents（AI 代理实例），最后由 Agents 调用 Tools（文件系统、浏览器、Cron 等）执行操作并返回结果。

这个架构设计解决了一个大问题：隐私和延迟。所有会话历史、凭证、甚至向量记忆都存储在 `~/.openclaw` 目录下，完全由用户掌控。报告中提到的会话隔离机制（`per-peer`、`per-channel-peer`）也让我印象深刻，它意味着我可以为不同的联系人配置完全不同的 AI 人格和工具权限，而不会互相污染。看完这部分，我心里有了底：这不是一个简单的聊天机器人包装器，而是一个完整的、可扩展的个人操作系统级平台。

## Fork 管理策略：在频繁迭代中守住生产环境的稳定性

搞清楚架构后，接下来的实际问题是怎么把代码跑起来。OpenClaw 目前还处于 pre-1.0 阶段，官方更新非常频繁。如果直接 `npm install -g`，一旦需要定制或者遇到 Bug，就只能被动等待官方修复，这显然不行。

我决定采用"Fork + 分层定制”的策略。在 Git 仓库里，我建立了三层分支架构：`upstream/main` 指向官方仓库，只负责定期 fetch；`origin/main` 保持与上游同步，但不包含我的任何修改；`origin/production` 才是我实际部署和运行的分支。

这个决策在今天 proved 非常正确。在首次部署时，我遇到了一个典型的 Windows 环境坑：执行 `pnpm build` 时报错，提示找不到 bash。排查发现是因为 Windows 默认的 `bash` 命令解析到了 WSL，而不是 Git Bash。解决方案是在项目根目录的 `.npmrc` 文件中添加 `script-shell` 配置，指向 Git 的安装路径。更重要的是，我用 `git update-index --skip-worktree .npmrc` 标记了这个文件，确保这个特定于我本地环境的配置不会被意外提交到仓库，污染了 Fork 的历史。这种“物理隔离”代码和“逻辑隔离”配置的思路，是长期维护这类快速演进项目的关键。

## TinyCore 极限部署评估：在 64MB 内存的边界上跳舞

既然架构这么轻量，我就忍不住想：它能不能跑在更极端的设备上？比如只有 64MB 内存的 Tiny Core Linux？于是我把目光投向了 `tinycore-feasibility-report.md`。

调研结果既让人兴奋又充满挑战。Tiny Core 的核心优势是极小（Core 版本仅 17MB），但它没有 systemd，包管理也是独特的 `.tcz` 格式。最大的拦路虎是 Node.js 版本：OpenClaw 要求 Node.js 22+，而 Tiny Core 官方仓库里的版本通常严重滞后。

报告里提出的“预编译打包 + 定制 Remaster"方案非常有启发性。我们不需要在 Tiny Core 上编译庞大的 Node.js，而是在开发机上裁剪掉不必要的渠道（只保留最轻量的 Telegram）、移除重型依赖（如 sharp、playwright），构建出一个精简版的 `openclaw-mini` 包。然后，通过 `ezremaster` 工具将这个包和预编译好的 Node.js 二进制文件直接打入 Tiny Core 的 ISO 镜像中。

虽然最终评估认为在如此受限的环境下开发效率会很低（评分 2/5），但对于嵌入式 IoT 或高安全要求的 Kiosk 场景，这种“极简内核 + 预编译依赖”的部署策略是完全可行的。这也让我意识到，OpenClaw 的模块化设计确实为这种极限裁剪提供了可能。

## 今天的收获

- 理解“本地优先 Gateway"作为核心枢纽连接多渠道与多工具的设计哲学，是进行高级定制和故障排查的前提。
- Fork 管理必须采用“三层分支隔离 + 选择性同步”策略，以平衡个人定制需求与上游频繁演进之间的矛盾。
- 消息渠道部署需遵循“先决条件确认 - 差异化配置 - 状态验证”的标准化流程，严格区分 Token 型与扫码型渠道的陷阱。
- AI 模型配置需严格区分“提供商认证机制”与“多 Agent 路由逻辑”，明确 API Key 与 OAuth 订阅的边界。
- Doctor 诊断工具是保障系统健康与配置规范化的核心防线，具备自动修复与深度审计能力，应定期运行。
- 在极度受限硬件（如 Tiny Core）部署时，需遵循“极简内核 + 预编译依赖 + 定制化 Remaster"策略，牺牲部分开发效率换取资源可行性。
