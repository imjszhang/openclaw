# 给 Agent 装上“自动进化”的引擎与合规的“外骨骼”

> Day 7 · 2026-02-06 · 骨架 [KL04](../../../pyramid/structure/P23-practice-diary/tree/KL04-2026-02-06.md)

昨天还在纠结怎么让 Agent 更聪明地回复消息，今天决定把视野拉高一点：如果 Agent 能自己发现不足并改进，甚至能合法地调用外部脚本突破沙箱，那它就不再是个只会听令的“工具”，而像个真正的“伙伴”了。今天的核心任务就是搭建这套自我进化机制，并打通外部脚本调用的任督二脉。

## 从“被动执行”到"OADA 闭环”的惊险一跃

早上我开始设计 Agent 的自我进化系统。起初我想沿用简单的“规划 - 执行 - 跟踪”三阶段模型，但很快发现这有个致命弱点：每一步都需要人工介入去执行具体的改进操作，这根本算不上“自主”。

为了解决这个问题，我引入了 **OADA (Observe-Analyze-Decide-Act)** 闭环。这个设计的核心难点在于如何让 Agent 安全地“自我修改”。我在 `agent-evolution-guide.md` 里反复推敲 `ACT` 阶段的逻辑：Agent 不仅要能修改发布计划或更新回复模板，还必须具备“后悔药”机制。

于是我强制加入了一套严格的安全约束：所有修改前必须自动备份，且备份文件带上时间戳，绝不覆盖原文件；同时支持一键回滚到任意历史版本。我还特意设计了“试运行模式”（dry-run），让 Agent 在真正动手前先预览修改内容并评估风险。这就好比给一个刚拿到驾照的新手配了个副刹车，只有当它连续几次“试驾”都表现良好，我们才敢在“成熟期”放开让它自动执行大部分低风险操作。这种渐进式的信任建立过程，比一开始就全权委托要稳妥得多。

## 打破沙箱：用外部脚本给 Agent 装上“外骨骼”

下午的重头戏是解决 Agent 的能力边界问题。Agent 再聪明，如果被锁在沙箱里，干不了调用系统命令、发复杂通知这些脏活累活，那也是白搭。我需要一条合规的路径让它能调用外部脚本。

翻阅 `external-scripting-guide.md` 时，我发现了几个宝藏接口。最直接的是 CLI 命令 `openclaw agent`，它简直就是为了 cron 系统量身定做的。我试着写了一个脚本，用 `openclaw agent --agent ops --message "生成周报" --deliver --reply-channel slack` 这条命令，成功让 Agent 在后台跑完分析后，直接把结果甩到了 Slack 频道里。

但更让我兴奋的是 HTTP Webhook 接口 `POST /hooks/agent`。这玩意儿太灵活了，我可以在 GitHub Actions 里配个流程，一旦有 PR 提交，就 curl 一下这个接口，带上 `sessionKey` 保持上下文，让 Agent 自动去审查代码。为了验证安全性，我特意配置了独立的 `hook token`，和 Gateway 的主认证分开管理。这样一来，外部脚本就像给 Agent 装上了合规的“外骨骼”，既扩展了能力，又没破坏核心的安全隔离。以前觉得 Agent 只能陪聊，现在它能帮我盯监控、审代码、发告警，这才是我想要的生产力工具。

## 今天的收获

- 真正的自主进化不是简单的任务循环，而是包含观察、分析、决策、执行、验证的 OADA 完整闭环。
- 安全机制（备份、回滚、试运行）是赋予 Agent 自我修改权限的前提，信任是赢得的而非给予的。
- 外部脚本集成（CLI 与 Webhook）是 Agent 突破沙箱边界、实现复杂自动化任务的合规路径。
- 会话隔离（sessionKey）能让外部触发器与主会话解耦，避免上下文污染的同时保留多轮对话能力。
- 渐进式实施策略（从人工审批到自动执行）是降低自主系统风险、建立人机信任的关键。
