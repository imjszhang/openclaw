# 让 AI 接管我的 Chrome：Browser Relay 实战手记

> Day 12 · 2026-02-11 · 骨架 [KL06](../../../pyramid/structure/P23-practice-diary/tree/KL06-2026-02-11.md)

昨天还在跟隔离的托管浏览器死磕，今天终于要动真格的了：让 AI 直接操作我日常用的 Chrome，继承所有登录态，这才是真正的“智能助手”该有的样子。

## 给 Chrome 装上“遥控器”

之前的托管浏览器虽然安全，但每次都要重新登录，做个自动化任务还得先过验证码，太折腾人。今天的核心任务就是部署 Browser Relay，让 AI 能直接控制我现有的标签页。

安装过程比想象中顺滑，一条命令 `openclaw browser extension install` 就把扩展文件扔到了 `~/.openclaw/browser/chrome-extension`。接下来就是标准的 Chrome 开发者流程：打开 `chrome://extensions`，开启开发者模式，加载已解压的扩展程序。

最让我惊喜的是它的“徽章”机制。加载完后，扩展图标上并没有立刻显示状态，只有当我手动点击图标附加当前标签页时，徽章才变成红色的 `ON`。这意味着 AI 不会莫名其妙地接管我正在看的页面，控制权完全在我手里。不过我也踩了个小坑：一开始图标显示红色的 `!`，排查半天发现是 Gateway 没起全，本地中继服务器（默认端口 18792）没跑起来。用 `curl -I http://127.0.0.1:18792/` 测了一下不通，重启 Gateway 后立马变绿。这种可视化的反馈机制，比看枯燥的日志直观多了。

## 智能体终于“懂”我的登录态了

装好扩展只是第一步，真正的魔法发生在智能体第一次调用 `browser` 工具时。

以前用托管浏览器，我得把 cookies 导来导去，或者让智能体在空白页重新登录。今天我只需在对话里说一句：“帮我在已打开的 GitHub 页面上操作”，然后确保目标标签页的徽章是 `ON`。智能体自动识别并使用了 `profile="chrome"` 参数，直接通过 CDP 协议向我的浏览器发号施令。

看着 AI 在我已经登录的 GitHub 页面上自动点击、输入，甚至读取页面内容，那种“它真的在用我的浏览器”的感觉非常强烈。架构图上画的那条链路——智能体 → Gateway → 本地中继 → 扩展 → 标签页，此刻在现实中跑通了。我也试了试右键菜单里的 "Attach all tabs in this window"，瞬间整个窗口的标签页都纳入了管控范围，新建标签页甚至会自动附加，这种无感知的接入体验，才是生产力工具该有的样子。

当然，权力越大责任越大。文档里反复强调的安全警告让我背脊发凉：一旦附加，AI 就能读取该标签页的所有内容，包括敏感信息和 cookies。所以我特意只附加了工作相关的标签页，操作完立刻右键 "Detach all tabs" 断开连接。这种“用完即走”的谨慎，可能是使用 Browser Relay 的必备素养。

## 今天的收获

- Browser Relay 通过本地中继服务器（端口 18792）和 Chrome 扩展协作，实现了非侵入式的标签页控制。
- 徽章状态（ON/!/...）是判断中继连接和附加状态最直观的调试依据，比查日志高效得多。
- 智能体操作的是“已附加”的标签页而非“当前激活”的标签页，明确控制边界能避免误操作。
- 继承现有浏览器的登录态极大提升了自动化效率，但也带来了读取敏感数据的安全风险，需最小化附加范围。
