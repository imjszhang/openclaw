# 终结手动换算：JS VI 系统内建微信封面自动化闭环

> Day 49 · 2026-03-20 · 骨架 [KL31](./KL31-2026-03-20.md)

今日主线任务是将 js-vi-system 的视觉能力延伸至微信公众号场景，解决运营团队长期面临的“手动换算像素”与“品牌视觉割裂”痛点，实现从 Prism 内容定稿到封面生成的全链路自动化。

## 确立微信专用尺寸枚举与静态约束

在之前的通用海报体系中，我们仅有 `a4`、`square` 等标准尺寸，面对微信公众号头条封面（约 900×383，2.35:1）和转发略缩图（500×500，1:1）时，运营人员或 Agent 往往需要自行换算比例，这不仅效率低下，更导致品牌一致性难以把控。为此，我直接在 `templates/_shared/sizes.js` 的全局 `SIZES` 枚举中新增了 `wechat-cover` 和 `wechat-thumb` 两个专用键。这一决策消除了所有中间换算环节，现在无论是通过 CLI 参数 `--size wechat-cover` 还是 OpenClaw 插件调用，系统都能直接锁定符合平台规范的像素约束，从源头确保了视觉输出的标准化。

## 构建独立模板目录与结构化元数据支持

为了适配微信后台对静态位图的强依赖，我构建了独立的 `templates/wechat-cover/` 目录，内含 `render.js`、`meta.json` 及 `styles.css`。在元数据设计上，我特意定义了 `title`、`subtitle`、`tag`、`issue` 等结构化字段，以便直接填入文章标题、栏目及期数信息。关键的技术决策在于显式禁用了动画支持（`animation.supported: false`），强制模板进入静态渲染模式。这确保了 `renderToHTML` 在后续处理时能正确内联样式，完美适配 PNG/JPEG 的直接导出需求，避免了因动效残留导致的渲染异常。

## 实现 CLI 与插件双模调用的自动化闭环

为了让这套新规范真正融入工作流，我实现了“独立 CLI"与"OpenClaw 插件”双模调用入口。在仓库内，开发者可直接执行 `node bin/js-vi.js poster -t wechat-cover --size wechat-cover --scheme dark -f png` 快速生成素材；而在 OpenClaw 体系中，`vi_poster_generate` 工具现已支持指定 `template: wechat-cover` 及相关尺寸参数。这种双模设计使得知识在 Prism 侧定稿后，Agent 能立即调用同一套品牌 Token（如 `daylight` 或 `dark` 方案）生成封面，彻底解决了“文章在 Prism、配图在别处”的割裂感，为后续 Cron 流水线中的无人值守发布奠定了基础。

## 明确 Puppeteer 依赖与运营职责边界

在交付环节，我进一步厘清了技术系统与运营操作的边界。导出 PNG/JPEG/PDF 强依赖本机安装的 Chrome 或 Edge 浏览器（通过 `puppeteer-core` 驱动），而 HTML 格式则无此依赖，非常适合用于快速校对版式。我在文档中明确界定：视觉系统仅负责像素精度与版式的一致性，至于微信后台对图片体积的压缩策略，则交由运营人员在编辑器内后续处理。此外，针对 `renderToHTML` 的逻辑，我修复了样式内联的隐性 Bug，确保 `meta._dir` 能作为 `templateDir` 正确传入，使得外部自定义模板也能准确加载 CSS，并通过 `package.json` 的 `exports` 字段开放了共享静态资源的引用路径。

## 今天的收获

- **场景化尺寸枚举**：将业务场景（如微信封面）直接映射为代码层面的尺寸键（`wechat-cover`），可彻底消除人工换算误差。
- **静态渲染优先**：针对特定交付渠道（如公众号后台），在模板元数据中显式禁用动画，能避免不必要的渲染复杂性并确保位图导出稳定。
- **双模调用架构**：同时提供 CLI 脚本与 Agent 插件入口，既能满足开发者调试需求，又能无缝嵌入自动化工作流。
- **职责边界清晰化**：明确系统只负责“像素与版式”，将“体积压缩”等非核心视觉逻辑剥离给下游运营环节，降低系统耦合度。
- **样式内联机制**：通过传递 `templateDir` 确保外部模板的 CSS 能被正确读取和内联，是支持模板扩展性的关键细节。
