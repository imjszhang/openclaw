# 消除人工换算：JS VI 系统内建微信封面图自动化闭环

> Day 49 · 2026-03-20 · 骨架 [KL31](./KL31-2026-03-20.md)

今日主线任务是解决公众号运营中的“视觉割裂”痛点：此前我们的通用海报尺寸缺乏微信专用命名，导致每次生成封面都需要人工换算 900×383 或 500×500 的像素比例，极易损害品牌一致性。我决定在 `js-vi-system` 中直接内建微信专用模板与尺寸枚举，实现从 Prism 内容定稿到封面生成的无缝自动化。

## 定义专用尺寸枚举与静态约束

针对微信平台特殊的像素规范，我首先在 `templates/_shared/sizes.js` 的全局 `SIZES` 枚举中新增了 `wechat-cover` (900×383) 和 `wechat-thumb` (500×500) 两个键。此前团队只能使用 `a4` 或 `square` 等通用尺寸，运营人员必须自行计算长宽比，这不仅效率低下，还容易因四舍五入导致图片在微信信息流中被裁剪。现在，通过 CLI 参数 `--size wechat-cover` 即可直接锁定标准规格。同时，为了适配微信后台对静态图片的偏好，我在元数据层面明确了该场景下禁用动画的约束，确保输出结果直接可用，无需二次加工。

## 构建独立模板目录与结构化元数据

为了支撑上述尺寸，我构建了独立的 `templates/wechat-cover/` 目录，其中包含 `render.js`、`meta.json` 及 `styles.css`。在 `meta.json` 中，我预定义了 `title`、`subtitle`、`tag`、`issue` 等结构化字段，这意味着 Agent 可以直接将文章标题和栏目期数填入模板，而无需硬编码字符串。关键的技术决策在于将 `animation.supported` 设为 `false`，强制进入静态渲染模式，这不仅简化了 `renderToHTML` 的逻辑，还确保了样式能正确内联。此外，我修复了一个隐性 Bug：`renderToHTML` 现在会将 `meta._dir` 作为 `templateDir` 传入 `wrapHTML`，确保即使使用外部扩展模板，其专属的 `styles.css` 也能被准确读取并打包进最终 HTML，避免了“只有内建模板有样式”的尴尬。

## 双模调用入口与工作流集成

为了让这套能力真正融入自动化闭环，我提供了 CLI 命令行与 OpenClaw 插件双模调用入口。在独立 CLI 模式下，运行 `node bin/js-vi.js poster -t wechat-cover --size wechat-cover --scheme dark -f png` 即可瞬间产出深色模式的封面图；而在 OpenClaw 生态中，`vi_poster_generate` 工具现已支持指定 `template: wechat-cover` 参数。这一改进的核心价值在于工作流的整合：当知识在 Prism 侧定稿后，Agent 可以立即调用同一套品牌规范（Neo-Brutalism + Cyberpunk Token）生成配图，彻底消除了“文章在 Prism、配图在 Photoshop"的割裂感，为后续 Cron 定时任务自动发布奠定了基础。

## 明确位图导出依赖与职责边界

在交付环节，我明确了基于 Puppeteer 的位图导出依赖与 HTML 校对流程的边界。生成 PNG/JPEG/PDF 强依赖本机安装的 Chrome 或 Edge 浏览器（通过 `puppeteer-core` 调用），这在无头服务器环境中需额外配置 `browserPath`；而 HTML 格式则无此依赖，非常适合在生成位图前进行快速的版式校对。我也划清了系统职责：`js-vi-system` 仅负责确保像素精确与版式符合品牌规范，至于微信后台对图片体积的压缩要求，则交由运营人员在编辑器内后续处理，不再试图在渲染引擎层过度优化压缩算法，保持了关注点分离。

## 今天的收获

- **尺寸枚举前置化**：将业务场景（如微信封面）的像素规范直接固化为代码中的 `SIZES` 枚举键，消除运行时的人工换算误差。
- **模板目录隔离**：为特定场景建立独立的模板目录（含 CSS/JS/Meta），并通过 `meta._dir` 动态注入样式，解决了外部模板样式丢失的问题。
- **静态渲染约束**：针对特定交付渠道（如微信后台），在元数据层直接禁用动画支持，从源头简化渲染管线并提升产出稳定性。
- **职责边界清晰化**：严格区分“视觉规范生成”与“运营压缩处理”，系统只保证像素与版式正确，不越界处理文件体积优化。
