# 断档八天后，我用 3600 次提交换了一套知识骨架

> Day 23 · 2026-02-22 · 骨架 [KL09](../../../pyramid/structure/P23-practice-diary/tree/KL09-2026-02-22.md)

距离上次记录已经过去了整整 8 天，这期间代码库像发酵的面团一样膨胀了不少。今天的主要任务很明确：把落后上游的 `githubforker` 分支强行拉齐，顺便把这段时间积累的一堆杂乱笔记理出个头绪来。

## 3600 次提交的“暴力”合并

打开终端执行合并时，心里其实有点打鼓。这次不是小打小闹的 patch，而是涵盖了 3600+ 个上游提交的大版本升级，横跨安全加固、功能新增、架构重构等五大领域。

冲突比预想的少，只有 5 个文件，但每一个都挺致命。比如在 `bash-tools.exec-runtime.ts` 和 `pi-tools.ts` 里，我之前为了图方便写的一些自定义 shell 配置逻辑，直接被上游更严谨的架构重构给覆盖了。看着那些被标记为 "ours" 的代码行，说实话有点肉疼——那是之前调试了好久的成果。

但这次我忍住了“魔改”的冲动。面对上游在类型提取和模块化上的大动作（比如把内联的 `ExecToolDefaults` 拆分成独立模块），我选择了全盘采纳 main 分支的版本。策略很明确：在架构演进的大方向面前，局部的定制化功能如果成了阻碍，那就先砍掉。我在合并总结里特意记了一笔：`shell/shellArgs` 功能丢失，后续需基于新架构重新实现。这种“先接纳、后重建”的策略，虽然当下痛苦，但避免了把 fork 变成一座无法维护的孤岛。

## 给知识安个家：三层分离的顿悟

合并完代码，转头面对自己这 20 多篇按日期堆砌的笔记，突然感到一阵窒息。读者（包括未来的我自己）根本不知道先读哪篇，宏观的项目分析和微观的终端配置混在一起，就像把砖头和图纸堆在同一个房间里。

这次大合并让我深刻意识到，代码需要架构，知识更需要。我花了一下午重新设计了目录结构，确立了 `journal` → `pyramid` → `outputs` 的三层分离方案。

最关键的决策是把 `tutorial`（教程）从顶层挪到了 `outputs` 目录下。之前我一直把教程和原始笔记、分析过程平级存放，现在想来大错特错。教程只是结构化思考后的一个“输出产物”，它不应该和“处理过程”混为一谈。现在的结构很清晰：`journal` 只管按时间记录原始素材，`pyramid` 负责自下而上地拆解和重组知识，而 `outputs` 则面向读者产出最终内容。为了应对未来的增长，我还把 `atoms` 按月分目录，把 `tree` 升级为目录结构，确保这套体系不会因为笔记变多而崩塌。

## 今天的收获

- 大规模合并 fork 时，应优先采纳上游的架构方向，即使这意味着暂时丢失局部定制功能。
- 被覆盖的功能必须文档化丢失细节和重建路径，避免知识黑洞。
- 知识库必须严格分离输入（journal）、处理（pyramid）和输出（outputs），职责单一才能扩展。
- 教程是思考的产出物，不应与思考过程（拆解体系）平级存放。
- 会随时间增长的产物（如 atoms、tree）必须用目录结构，静态产物（如 SCQA）保持单文件。
