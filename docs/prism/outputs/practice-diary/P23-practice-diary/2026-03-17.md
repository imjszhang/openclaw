# 从 Docker 依赖到插拔式架构：第 46 天大规模上游合并实战

> Day 46 · 2026-03-17 · 骨架 [KL28](./KL28-2026-03-17.md)

今日主线任务是将 `origin/main` (commit `4649f82b77`) 的 143 个上游提交合入本地 `githubforker` 分支，核心挑战在于处理约 4300 个文件的变更，并借此完成从强依赖 Docker 向插拔式 Sandbox 架构的彻底转型。

## 执行“全量同步 - 冲突主分支优先”策略，完成架构转型

本次合并涉及自共同祖先 `389647157` 以来的巨大跨度，最终合并提交定为 `2b0d107de4`，仓库版本演进至 `2026.3.14`（目前位于 CHANGELOG 的 Unreleased 段）。在合并过程中，我遭遇了 7 个文件冲突，包括 `pnpm-lock.yaml`、`src/agents/transcript-policy.ts` 以及已被移除的 Chrome 扩展相关资源（`assets/chrome-extension/*` 等）。面对这些冲突，我严格遵循“主分支优先”原则，直接采纳 Main 分支版本，特别是针对 Breaking Change 中移除旧版 Chrome 扩展中继路径的改动，果断删除了本地遗留的 `driver: "extension"` 配置，确保后端执行环境成功重构为支持 SSH 与远程镜像的插拔式架构。这一决策彻底解除了对单一 Docker 模式的强依赖，新增的核心 SSH 后端现在支持基于 secret 的 key、证书与 known_hosts 配置，为多环境部署奠定了物理基础。

## 通过捆绑插件机制统一路径，实现秒级冷启动

为了解决规模化带来的启动负载问题，本次更新将 OpenRouter、GitHub Copilot 以及 MiniMax API 的逻辑全部收敛至默认开启的捆绑插件中。我确认了新的优先级规则：显式安装的插件在 duplicate-id 时胜出，且捆绑插件默认优于自动发现的全局插件。配合安装器直连 GitHub main 分支的能力（支持 `openclaw update --tag main`），以及 Gateway 改为从编译后的 `dist/extensions` 直接加载捆绑渠道插件，实测 WhatsApp 等渠道的冷启动时间已压缩至秒级。这一改动不仅消除了每次启动重编 TypeScript 的开销，还确立了安装器直连与秒级响应的交付标准，大幅降低了部署时的资源等待成本。

## 深化多渠道功能增强，构建结构化审批与容错体系

在业务交互层，我重点验证了 Feishu 和 Telegram 渠道的结构化升级。Feishu 现在支持结构化审批卡片，流式思考 token 能以 blockquote 形式优雅展示，且当前会话 ACP 与 subagent 会话实现了精准绑定；Telegram 则新增了默认关闭的 `channels.telegram.silentErrorReplies` 配置，允许错误回复静默发送，避免打扰用户。针对 Skills 提示预算超限的问题，系统现在会自动回退为紧凑目录而非直接丢弃，保障了功能可见性。同时，针对 WhatsApp 重连时序导致的 Baileys 515 配对问题，通过增加新近度过滤与 protobuf Long 时间戳处理完成了修复，并移除了不稳定的 Chrome 扩展中继路径，引导用户通过 `openclaw doctor --fix` 迁移至 `existing-session` 模式，确保了多端交互的长期稳定性。

## 收紧权限校验与身份绑定，确立“故障即关闭”底线

安全是本次合并的兜底策略。我强制实施了插件 Context Engine 与所有者的绑定注册，从根本上防止了特权伪造或通过 SDK 直接覆盖已有 engine id 的风险。在 Mattermost 和 Google Chat 等渠道，发送方校验逻辑被进一步收紧，Twitch 的空 allowlist 现在被明确视为全部拒绝。对于 ACP 审批流程，一旦存在冲突提示，系统严格执行"fail closed"策略，且变更内部 action 必须拥有 admin 范围。此外，针对设备配对失败的场景，公开返回的信息已被泛化处理，内部具体原因仅记录在日志中，配合固定的 Webhook 路由表，构建了从身份认证到数据传输的纵深防御网，杜绝了因内部实现细节泄露导致的安全隐患。

## 今天的收获

- **架构解耦**：通过引入 SSH 与 OpenShell 后端，成功将执行环境从 Docker 强依赖中解耦，实现了真正的插拔式 Sandbox 架构。
- **启动优化**：利用捆绑插件直连 `dist` 目录与 main 分支安装策略，将网关冷启动从分钟级优化至秒级，显著降低部署负载。
- **安全底线**：确立"fail closed"审批策略与身份强绑定机制，在规模化自动化场景中有效阻断了特权伪造与内部信息泄露风险。
- **冲突治理**：在大规模合并（143+ 提交）中，坚持“主分支优先”原则处理 Breaking Change，是保持版本演进一致性的最高效手段。
