# 从 Docker 依赖到插拔式架构：第 46 天的大规模上游合并实战

> Day 46 · 2026-03-17 · 骨架 [KL28](./KL28-2026-03-17.md)

今日的核心任务是执行一次高风险的大规模上游合并，将 `origin/main` (commit `4649f82b77`) 的 143 个提交合入我们的 `githubforker` 分支。面对约 4300 个文件的变更和后端执行环境的根本性重构，我们必须确保在解除 Docker 强依赖的同时，维持版本演进（2026.3.14）的绝对一致性。

## 执行“全量同步 - 冲突主分支优先”策略，完成架构转型

本次合并不仅仅是代码量的堆叠，更是架构范式的转移。上游引入了可插拔的 Sandbox 后端，支持 OpenShell 的 `mirror` 与 `remote` 模式，并新增了基于 secret 管理的核心 SSH 后端，彻底打破了以往对单一 Docker 模式的依赖。在合并过程中，我遇到了 7 个关键文件冲突，包括 `pnpm-lock.yaml`、`src/agents/transcript-policy.ts` 以及涉及 Chrome 扩展中继路径的废弃文件（如 `assets/chrome-extension/*`）。面对这些冲突，我严格遵循“主分支优先”原则，直接采纳 Main 分支版本，特别是针对 Breaking Change 中移除旧版 Chrome 扩展中继路径的改动，果断删除了相关资产和文档，确保本地 browser 配置能顺利迁移至 `existing-session` 或 `user` 模式。最终通过 `git commit --no-verify` 完成了提交，避免了因参数列表过长导致的 pre-commit 钩子失败。

## 通过捆绑插件机制统一 Provider 与技能发现路径

为了解决冷启动慢和依赖混乱的问题，本次合并确立了“显式安装优于自动发现”的优先级规则。我们将 OpenRouter、GitHub Copilot 以及 MiniMax API（合并为默认开启的单一 `minimax` 插件）的逻辑全部收敛至捆绑插件中。这一改动配合安装器直连 GitHub `main` 分支的能力（支持 `openclaw update --tag main`），极大地简化了部署流程。最显著的效果体现在网关启动性能上：Gateway 现在直接从编译后的 `dist/extensions` 加载捆绑渠道插件，不再每次重编 TypeScript，使得 WhatsApp 等渠道的冷启动时间被压缩至秒级，大幅降低了部署负载。

## 深化多渠道功能增强与交互体验优化

在业务功能层，我们重点增强了结构化审批与自适应容错体系。在 Feishu 渠道，我们引入了结构化审批卡片，并实现了流式思考 token 以 blockquote 形式展示，提升了交互的透明度；Telegram 渠道则新增了 `channels.telegram.silentErrorReplies` 配置，允许错误回复静默发送，减少噪音。针对 Skills 提示预算超限的问题，系统现在会自动回退为紧凑目录，确保所有已注册 skills 依然可见而非直接丢弃。此外，针对 WhatsApp 重连时的时序问题，我们实施了新近度过滤与 protobuf Long 时间戳处理，修复了 Baileys 515 配对重启后的凭据写入竞争条件，确保了多端交互的稳定性。

## 收紧权限校验与身份绑定策略，确立安全底线

随着系统规模扩大，安全防御必须从“默认允许”转向“故障即关闭”。本次合并强制插件 Context Engine 与所有者绑定，从根本上防止了特权伪造或通过 SDK 直接覆盖已有 engine id 的风险。在 Mattermost 和 Google Chat 等渠道，我们收紧了发送方校验逻辑，并对 ACP 审批实施了严格规范：当存在冲突提示时，系统采取 `fail closed` 策略，且变更内部 action 必须拥有 admin 范围。针对设备配对失败的场景，我们泛化了公开返回的错误信息，仅将内部原因记录在日志中，构建了从身份认证到数据传输的纵深防御网，杜绝了内部实现细节的泄露。

## 今天的收获

- **冲突解决原则**：在大规模上游合并中，对于架构重构类冲突（如 Sandbox 后端切换、废弃文件删除），应坚决执行“主分支优先”策略，避免保留过时逻辑导致系统分裂。
- **启动性能优化**：通过将 Provider 逻辑收敛至捆绑插件并直接从 `dist` 目录加载，可消除运行时编译开销，将网关冷启动从分钟级优化至秒级。
- **安全防御底线**：在身份校验和审批流程中，必须预设 `fail closed`（故障即关闭）机制，并对错误信息进行泛化处理，防止内部逻辑泄露。
- **配置迁移自动化**：针对 Breaking Change（如 Chrome 扩展路径移除），需在合并文档中明确提供 `openclaw doctor --fix` 等自动化迁移指引，降低用户升级成本。
