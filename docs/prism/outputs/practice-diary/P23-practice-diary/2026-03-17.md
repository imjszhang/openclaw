# 从 Docker 依赖到插拔式沙箱：第 46 天的大规模上游合并实战

> Day 46 · 2026-03-17 · 骨架 [KL28](../key-lines/KL28-2026-03-17.md)

今日主线任务是将 `origin/main`（commit `4649f82b77`）合入我们的 `githubforker` 分支，这是一次涉及 143 个上游提交、约 4300 个文件变更的“大手术”。面对如此体量的代码洪峰，核心挑战在于如何在确保版本演进（2026.3.14）完整性的同时，彻底完成从单一 Docker 依赖向插拔式 Sandbox 架构的转型，并守住安全底线。

## 执行“全量同步 - 冲突主分支优先”策略，完成架构转型

本次合并的核心决策是严格执行“主分支优先”原则。在解决 7 个具体文件冲突时，包括 `pnpm-lock.yaml`、`src/agents/transcript-policy.ts` 以及被移除的 Chrome 扩展相关资源（`assets/chrome-extension/*` 等），我们全部采用了 Main 分支的版本。这一策略直接促成了后端执行环境的重构：我们不再强依赖单一的 Docker 模式，而是引入了支持 SSH 与远程镜像的插拔式架构。新增的核心 SSH 后端支持基于 secret 的 key、证书与 `known_hosts` 配置，而 OpenShell 则专注于生命周期管理与可选的 `mirror` 模式。这种结构性的解耦，为后续支持 `remote` 工作区模式奠定了物理基础，标志着我们的沙箱体系真正具备了弹性。

## 通过捆绑插件机制统一路径，实现秒级冷启动

在部署优化层面，我们做出了一个关键改变：将 OpenRouter、GitHub Copilot 以及 MiniMax API 的 Provider 逻辑全部收敛至默认开启的捆绑插件中。这一举措确立了“显式安装优于自动发现”的优先级规则，有效避免了插件 ID 冲突。更显著的效果体现在启动速度上：Gateway 现在直接从编译后的 `dist/extensions` 目录加载捆绑渠道插件，不再每次启动都重编 TypeScript。实测中，WhatsApp 类渠道的冷启动时间已被压缩至秒级，大幅降低了部署负载。同时，安装器现在支持直连 GitHub `main` 分支（通过 `openclaw update --tag main`），配合编译后目录加载机制，让交付效率有了质的飞跃。

## 深化多渠道交互体验，构建自适应容错体系

业务层的增强主要集中在多渠道的结构化审批与容错能力上。在 Feishu 渠道，我们引入了结构化审批卡片，并将流式思考 token 以 blockquote 形式展示，提升了交互的透明度；Telegram 则新增了默认关闭的 `channels.telegram.silentErrorReplies` 配置，允许错误回复静默发送，减少噪音。针对 WhatsApp 重连时的时序问题，我们实施了新近度过滤与 protobuf Long 时间戳处理，修复了 Baileys 515 配对重启后的凭据写入竞争。此外，针对 Chrome 扩展中继路径移除这一 Breaking Change，我们明确了迁移指引：本机 browser 配置需运行 `openclaw doctor --fix` 迁移至 `existing-session` 或 `user` 模式，确保了协议升级过程中的平滑过渡。

## 收紧权限校验与身份绑定，确立“故障即关闭”底线

安全加固是本次合并的兜底环节。我们强制插件 Context Engine 注册时必须与所有者绑定，从根源上防止了特权伪造或通过直接 SDK 导入覆盖已有 engine id 的风险。在 Mattermost 和 Google Chat 等渠道，我们收紧了发送方校验逻辑，并对 ACP 审批采取了严格的"fail closed"策略——即在存在冲突提示时直接拒绝，且变更内部 action 必须拥有 admin 范围。针对设备配对失败的场景，我们实施了信息泛化处理，公开返回信息中不包含内部原因，仅记录在日志中，构建了从身份认证到数据传输的纵深防御网，杜绝了因错误信息泄露导致的内部逻辑暴露。

## 今天的收获

- **架构解耦优先**：从单一 Docker 转向插拔式 Sandbox（SSH/OpenShell），不仅解决了环境依赖问题，更为远程执行和镜像模式提供了弹性基础。
- **启动性能优化**：通过“捆绑插件 + 预编译 dist 加载”替代运行时重编，将网关冷启动从分钟级压缩至秒级，是规模化部署的关键。
- **安全默认值原则**：在冲突审批、设备配对失败等边缘场景坚持"fail closed"和信息泛化，是防止内部逻辑泄露和权限逃逸的有效手段。
- **迁移指引明确化**：面对 Breaking Change（如 Chrome 扩展移除），提供明确的 `doctor --fix` 迁移路径比单纯的功能移除更能保障用户留存。
