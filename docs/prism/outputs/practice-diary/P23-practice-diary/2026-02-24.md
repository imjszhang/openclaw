# 把散乱笔记折射成光谱：知识棱镜工具诞生记

> Day 25 · 2026-02-24 · 骨架 [KL21](../../../pyramid/structure/P23-practice-diary/tree/KL21-2026-02-24.md)

前二十多天攒了二十多篇笔记，渠道部署、Agent 配置、安全加固……内容虽多，但想查个核心概念还得在五个文件里拼凑。今天决定不再只是记录，而是动手造个“棱镜”，把这一束散乱的白光，自动折射成面向读者的清晰光谱。

## 给笔记装上“只增不改”的过滤器

动手写代码前，最纠结的是架构怎么搭。以前的习惯是边记边改，结果越改越乱，原始探索过程全丢了。这次我狠下心定了一条铁律：**Journal 层（原始素材）必须“只增不改”**。

我把项目强行拆成了三层：`journal/` 负责忠实记录时间线，`pyramid/` 负责结构化拆解，`outputs/` 负责生成最终文章。为了落地这个想法，我在 `lib/process.mjs` 里写了一套三阶段增量管线：

1.  **Atoms 提取**：LLM 读取新增的 journal，提炼最小知识单元，按月归档。
2.  **Groups 归组**：跨文档把 atoms 聚拢，强行提炼出一个观点句。
3.  **Synthesis 收敛**：从所有观点里再收敛出顶层候选。

调试时差点踩坑，担心重复运行会产生重复数据。后来在管线里加了幂等性检查，确保同一篇 journal 无论跑多少次，生成的 atom 都是唯一的。看着散乱的 markdown 文件被自动拆解、归组，那种“秩序感”终于回来了。

## 零依赖核心与双入口的博弈

技术选型时，我差点习惯性地 `npm install` 一堆库。但转念一想，这是个知识管理工具，如果安装门槛太高，谁愿意用？于是做了一个反直觉的决定：**核心库和独立 CLI 坚持零外部依赖**。

除了打包技能包时用了个 `archiver`，核心逻辑全靠 Node.js 原生 `fetch` 和 ES Module 搞定。为了兼顾不同场景，我设计了“双入口”：

- **独立 CLI**：`npx js-knowledge-prism init`，不用装 OpenClaw 也能跑，配置走 `.env`。
- **OpenClaw 插件**：深度集成到 Agent 对话里，配置走 `openclaw.json`。

晚上 22:18 提交 Initial Commit 时，看着 `bin/cli.mjs` 和 `lib/` 目录下那几个干净的 `.mjs` 文件，心里挺踏实。没有复杂的依赖树，只有纯粹的逻辑流转，这才是工具该有的样子。

## 今天的收获

- 笔记的价值不在于记录了多少，而在于能否被低成本地检索和复用。
- “只增不改”的原始素材层是保证知识演进可追溯的基石。
- LLM 驱动的增量管线比规则匹配更能适应非结构化的思考过程。
- 零依赖核心策略能极大降低工具的传播门槛和生态兼容性风险。
- 双入口设计让同一套核心逻辑能同时服务于命令行极客和 Agent 用户。
