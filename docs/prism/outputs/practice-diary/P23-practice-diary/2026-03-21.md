# 上游 PR 分支隔离与插件架构解耦实战

> Day 50 · 2026-03-21 · 骨架 [KL32](./KL32-2026-03-21.md)

今天的核心任务是在向 `openclaw/openclaw` 上游提交 ACP 插件 API 改动时，彻底解决长期开发分支与官方基线的冲突难题，并同步完成插件层「类型契约」与「运行时装配」的物理拆分。面对 fork 定制线与上游 main 的日益 divergent，我强制执行了独立分支策略，并在代码层面通过 `types.ts` 和 `registry.ts` 的重构，为插件系统构建了更严密的安全防线。

## 上游 PR 的独立分支隔离与 Cherry-pick 回合并

在向官方上游提交 PR #51187 时，我严格遵循了「独立分支隔离」原则。首先执行 `git fetch source main`，基于纯净的 `source/main` 创建了 `feat/plugin-types-registry` 分支，专门用于承载 `src/plugins/types.ts` 中新增的 `registerAcpRuntimeBackend` 接口及 `src/plugins/registry.ts` 中的装配逻辑委托。这一决策避免了将我个人 fork 中 `js-clawhub` 长期线的私有定制（如特定的扩展 API 或实验性配置）污染到给上游的评审视野中。

然而，真正的挑战出现在将改动合回本地 `js-clawhub` 分支时。起初我尝试直接 `git merge feat/plugin-types-registry`，结果在 `src/extensionAPI.ts` 等与本次 ACP 改动无关的文件上爆发了大量的 add/add 冲突。这些冲突纯粹源于 fork 线与 upstream main 的自然分叉，毫无业务价值。我果断中止了合并操作（`git merge --abort`），转而使用 `git cherry-pick` 提取了那个包含插件改动的特定提交。这一操作仅将三个核心文件的变动精准落地到 `js-clawhub`，既保持了上游基线的纯净，又显著降低了合并成本，完美规避了「为了合一个功能而解决一堆无关冲突」的陷阱。

## 插件注册表架构的类型契约与运行时解耦

在代码内部，我实施了插件层架构的深度解耦，将原本模糊的插件定义拆分为 `src/plugins/types.ts`（类型契约声明）与 `src/plugins/registry.ts`（运行时装配逻辑）。在 `types.ts` 中，我明确了 `OpenClawPluginApi` 的完整接口，包括新增的 ACP runtime backend 注册方法，并定义了 `PluginRegistrationMode` 枚举（`full` / `setup-only` / `setup-runtime`）。这种声明式的设计确保了 loader、hooks 测试等模块仅需依赖类型定义，无需引入沉重的运行时逻辑，清晰了依赖边界。

而在 `registry.ts` 中，我构建了真正的「执行层」。`createPluginRegistry` 工厂函数负责收集 `PluginRecord`，并在加载时执行严格的去重与诊断逻辑。特别是在 HTTP 路由注册环节，系统现在会强制校验路径规范化及 `auth` 类型（必须为 `gateway` 或 `plugin`），若新路由与现有路由重叠且鉴权不一致，或试图抢占其他插件已占用的路由，注册将被直接拒绝。此外，针对 `suppressGlobalCommands` 为 true 的快照场景，命令注册被限制为仅校验并记入本地 registry，严禁写入全局命令表，从而防止了非激活状态下的副作用污染。

## 提示注入过滤与安全边界的精细化控制

本次重构的另一大收获是在安全层面的精细化控制，特别是针对提示注入（Prompt Injection）的防御。在 `types.ts` 中，我定义了 `PLUGIN_PROMPT_MUTATION_RESULT_FIELDS` 常量，明确标识出那些可能篡改 system prompt 或上下文的危险字段。配合 `registry.ts` 中的逻辑，当配置项 `allowPromptInjection` 设为 false 时，系统会对 `before_agent_start` Hook 的返回结果自动包裹一层 `constrainLegacyPromptInjectionHook`。

该机制会在同步或异步返回路径上调用 `stripPromptMutationFieldsFromLegacyHookResult`，强制剥离所有标记为危险的改写字段，仅保留模型覆盖或提供方切换等安全元数据。这种「类型定义 + 运行时过滤」的双重保障，确保了即使在插件作者误用旧版 Hook 返回格式时，系统也能自动阻断潜在的提示注入攻击，将安全防线从「依赖人工审查」推进到了「代码强制执行」的级别。

## 今天的收获

- **上游协作分支策略**：向官方提交 PR 必须基于 `source/main` 创建独立分支，合回本地长期线时严禁直接 Merge，应改用 `git cherry-pick` 提取特定提交以规避无关差异导致的高成本冲突。
- **架构分层原则**：插件系统应严格物理拆分为 `types.ts`（声明契约）与 `registry.ts`（执行装配），前者定义接口与模式枚举，后者负责去重、诊断及副作用控制，以清晰化依赖边界。
- **运行时安全防御**：通过 `PluginRegistrationMode` 控制不同场景下的 API 可用性，并利用 `stripPromptMutationFieldsFromLegacyHookResult` 在运行时强制剥离危险字段，实现针对提示注入的自动化过滤。
- **路由与命令隔离**：在注册 HTTP 路由和 CLI 命令时实施严格的冲突检测（如 auth 一致性检查、跨插件占用拦截），并在快照加载场景下通过 `suppressGlobalCommands` 标志位防止污染全局状态。
