# 上游 PR 分支隔离与插件架构解耦实战

> Day 50 · 2026-03-21 · 骨架 [KL32](./KL32-2026-03-21.md)

今天的核心任务是将 ACP 运行时后端接入插件 API，并同步完成向官方上游的 PR 提交与本地长期分支的合回。在操作过程中，我深刻体会到当 Fork 定制线偏离上游 Main 较多时，传统的 Merge 策略会带来巨大的冲突成本，必须通过精细的分支隔离与 Cherry-pick 策略来维持代码库的整洁与安全。

## 上游 PR 贡献必须采用「独立分支隔离 + Cherry-pick 回合并」策略

在向 `openclaw/openclaw` 提交关于 `registerAcpRuntimeBackend` 的 PR 时，我严格遵循了“独立分支隔离”原则。首先执行 `git fetch source main`，基于纯净的 `source/main` 创建了 `feat/plugin-types-registry` 分支，而非直接使用我本地的 `js-clawhub` 长期开发分支。这一决策确保了上游维护者评审的基线是最干净的官方 Main，避免了将我 Fork 中特有的定制逻辑（如 `js-clawhub` 线上的大量修改）混入评审视野。

代码改动集中在三个文件：在 `src/plugins/types.ts` 中扩展了 `OpenClawPluginApi` 类型定义，在 `src/plugins/registry.ts` 的 `createApi` 中委托注册逻辑，并在测试辅助文件中补全了空实现。提交信息定为 `plugins: wire ACP runtime backends into plugin API` 后，我将其推送到远端并成功发起了 PR #51187。

真正的挑战出现在将改动合回本地 `js-clawhub` 分支时。起初我尝试直接 `git merge feat/plugin-types-registry`，结果在 `src/extensionAPI.ts` 等与本次 ACP 功能完全无关的文件上爆发了大量的 `add/add` 冲突。这是因为我的长期分支与上游 Main 在其他维度已经产生了分叉。意识到这是无效冲突后，我立即执行 `git merge --abort` 中止操作，转而使用 `git cherry-pick` 仅提取包含插件改动的特定提交。这一操作成功将三处核心代码变动落到了 `js-clawhub` 上，同时完全规避了无关文件的冲突，显著降低了合并成本。此外，针对工作区存在未跟踪文件导致 `format:check` 失败的情况，我使用了 `git commit --no-verify` 快速完成了提交，确保了流程的顺畅。

## 插件注册表架构必须严格分离「类型契约声明」与「运行时装配逻辑」

在实现 ACP 后端接入的过程中，我再次确认了插件层架构中 `types.ts` 与 `registry.ts` 物理分离的必要性。`types.ts` 专注于声明契约，定义了 `OpenClawPluginApi` 的所有入口（如 `registerTool`, `registerHook` 等）以及 `PluginRegistrationMode` 枚举（包含 `full`, `setup-only`, `setup-runtime`）。这种设计让 loader 和 hooks 模块可以仅依赖类型定义，而无需卷入复杂的运行时逻辑，清晰了依赖边界。

`registry.ts` 则负责具体的装配过程与副作用控制。在 `createPluginRegistry` 中，我们根据 `registrationMode` 决定哪些注册方法是空操作，从而在 `setup-only` 模式下避免误触全局副作用。特别是在安全控制方面，架构实施了细粒度的权限与过滤机制：对于 HTTP 路由注册，系统会强制校验路径规范化及 `auth` 类型（必须为 `gateway` 或 `plugin`），若新路由与现有路由重叠且鉴权不一致，或试图抢占其他插件已占用的路由，注册将被直接拒绝。

更关键的是针对提示注入的防御。我们在 `types.ts` 中定义了 `PLUGIN_PROMPT_MUTATION_RESULT_FIELDS` 标识危险字段，并在 `registry.ts` 的 `registerTypedHook` 中实施了拦截逻辑。当配置 `allowPromptInjection === false` 时，系统会自动包裹 `before_agent_start` Hook，调用 `stripPromptMutationFieldsFromLegacyHookResult` 剥离可能修改 system prompt 的字段，仅保留模型覆盖等非改写结果。此外，在快照加载场景下，通过 `suppressGlobalCommands === true` 标志，命令注册仅校验并记入本地 registry，不写入全局命令表，有效防止了非激活插件污染正在运行的网关环境。

## 今天的收获

- **分支隔离策略**：向官方上游提 PR 必须基于 `source/main` 创建独立分支，严禁直接使用包含大量定制逻辑的长期开发分支，以保持评审基线纯净。
- **Cherry-pick 优于 Merge**：当 Fork 长期分支与上游 Main 存在较多无关差异时，合回特定功能提交应使用 `git cherry-pick` 而非 `git merge`，可避免大量无效的 `add/add` 冲突。
- **类型与逻辑物理拆分**：插件架构应将 `types.ts`（契约声明）与 `registry.ts`（装配逻辑）严格分离，使依赖边界清晰，便于模块按需引用。
- **运行时安全防线**：必须在注册表层面实施路由冲突检测、鉴权强制校验及提示注入字段剥离机制，构建防全局副作用误触的安全可控环境。
- **快照模式隔离**：利用 `suppressGlobalCommands` 标志在快照加载场景下将命令注册限制在本地 registry，避免污染全局命令空间。
