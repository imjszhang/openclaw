# 拒绝重型框架：用原生 JS 和 Neo-Brutalism 一天搞定生态导航站

> Day 13 · 2026-02-12 · 骨架 [KL20](../../../pyramid/structure/P23-practice-diary/tree/KL20-2026-02-12.md)

昨天还在纠结 OpenClaw 生态里那些散落的宝藏项目怎么被用户发现，今天决定不再空想，直接动手把"JS ClawHub"这个策展型导航站从概念变成现实。既然官方 ClawHub 负责“安装”，那我们就负责“发现”，目标很明确：零后端、纯静态、一天内让首版跑起来。

## 砍掉 React，拥抱原生 HTML 的“暴力美学”

项目一开始，最大的诱惑就是顺手开个 React 或 Vue 项目，毕竟组件化开发太香了。但转念一想，这只是一个内容导航站，引入构建工具链和虚拟 DOM 简直是杀鸡用牛刀，还会徒增加载体积。于是我做了一个大胆的决定：回归原生 HTML + 模块化 JS。

为了不让页面看起来像 90 年代的 GeoCities 遗址，我选择了 Neo-Brutalism（新粗野主义）风格。这种风格大胆、高对比度，正好契合“策展”的个性。配合 Tailwind CSS 快速搭建响应式布局，再用 Three.js 在首页甩出一个旋转的 3D 背景，沉浸感瞬间拉满。

最让我满意的是数据与展示的彻底解耦。我没有把内容硬编码在 HTML 里，而是全部扔进 JSON 文件，写了一个轻量级的渲染逻辑：页面加载时读取 JSON，动态生成卡片。这样做的好处是，后续如果要加新模块，只需改数据源，模板几乎不用动。这种“极简架构”让我在下午 19:45 就完成了 Initial Commit，基础结构和首页竟然在几小时内就立住了。

## 自研 I18n 与 CLI 自动化：把重复劳动交给脚本

导航站要想有国际范儿，双语支持是必须的。市面上那些庞大的 i18n 库对我来说又太重了。我手写了一个 `I18nManager`，核心逻辑简单粗暴：给所有需要翻译的 DOM 元素打上 `data-i18n` 属性，启动时根据 `localStorage` 里的偏好，去 `locales/zh-CN.js` 或 `en-US.js` 里捞文案替换。对于 Markdown 文章，则通过检测 `.en-US.md` 后缀来加载对应版本。这套方案虽然土，但胜在可控，且没有任何运行时开销。

到了部署环节，我实在不想每次改个文案都要手动 `git add`、`commit`、`push` 还要检查 GA 代码有没有注入。于是顺手写了个 `cli/cli.js`。最爽的时刻是敲下 `node cli/cli.js sync` 的那一刻：脚本自动执行了文件复制、Google Analytics 注入、i18n 完整性校验、Pulse 数据清理、API 生成等 7 个步骤，最后自动提交并推送到 GitHub Pages。

看着终端里滚动的日志，从构建完成到 Cloudflare CDN 生效，整个闭环无人值守。这种“把流程固化成命令”的快感，比写业务代码本身还让人上瘾。

## 今天的收获

- 对于内容型导航站，原生 HTML 加模块化 JS 往往比重型框架更轻量且维护成本更低。
- 架构上严格分离源码（src）与部署产物（docs），是保证构建管线清晰可控的前提。
- 轻量级的自研 i18n 方案（data-i18n + JSON）足以应对中小型项目的多语言需求，无需引入庞大依赖。
- 将构建、校验、部署等重复劳动封装进 CLI 命令，是实现“无人值守”迭代的关键。
- 明确的差异化定位（人工策展 vs 自动聚合）比技术堆叠更能决定产品的核心价值。
