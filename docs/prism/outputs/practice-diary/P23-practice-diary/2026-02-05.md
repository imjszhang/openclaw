# 给 Agent 装上"生物钟"：Cron 与心跳的选型博弈

> Day 6 · 2026-02-05 · 骨架 [KL03](../../../pyramid/structure/P23-practice-diary/tree/KL03-2026-02-05.md)

昨天让 Agent 学会了"被动响应"，今天得给它装个"生物钟"，让它能在没人喊的时候自己动起来。原本以为只是配个定时任务那么简单，结果在配置过程中发现，OpenClaw 里竟然藏着两套完全不同的自动化机制，选错了不仅任务跑不起来，还可能把主会话搞得一团糟。

## 差点把主会话搞成"垃圾场"的 Session Target

上午我想给 Agent 加个每 5 分钟执行一次脚本的任务，顺手敲了条命令想让它定期跑个 `python workflow.py`。起初我理所当然地觉得，任务嘛，就在主会话里跑呗，反正都是同一个 Agent。

结果配置完跑了几次，发现主会话里全是脚本执行的日志和工具调用的中间状态，原本清爽的对话上下文被污染得没法看。更糟糕的是，有次我想跟 Agent 聊点别的，它居然还沉浸在刚才脚本执行的上下文里，回答牛头不对马嘴。

翻文档才发现罪魁祸首是 `sessionTarget` 配成了 `main`。原来 OpenClaw 的 Cron 任务有两种运行空间：`main` 是跟咱们日常对话共享上下文的，适合做提醒；而 `isolated` 才是给后台自动化任务准备的，每次运行都是个全新的临时会话，ID 格式是 `cron:<jobId>`，跑完就扔，绝不污染主对话。

赶紧用 `openclaw cron edit` 把任务改成了 `--session isolated`，世界瞬间清净了。这也让我明白一个硬规则：只要是想让 Agent 执行 shell 命令、跑脚本或者做后台同步，必须无脑选 `isolated`；只有那种"20 分钟后提醒我喝水"的简单通知，才配用 `main`。

## Cron 的"精确打击"与心跳的"模糊感知"

解决了会话污染问题后，我又纠结起另一个事：有些任务到底该用 Cron 还是用 Heartbeat？

之前看文档以为 Heartbeat 就是个简化版的 Cron，反正都是定时跑。今天深入对比才发现，这俩简直是两种完全不同的思维模式。

Heartbeat 像是个"模糊感知"的管家，默认每 30 分钟（OAuth 模式是 1 小时）醒来一次，拿着 `HEARTBEAT.md` 清单扫一眼："收件箱有急事吗？"、"两小时内有会吗？"。如果没事，它回个 `HEARTBEAT_OK` 就静默丢弃，根本不打扰你；如果有事，才投递消息。它的优势是省资源，一次调用能覆盖多项检查，而且不会让会话因为心跳而"假性活跃"。

但 Cron 完全不同，它是"精确打击"的特种兵。你可以用 cron 表达式精确到"每周一早上 9 点"，或者用 `--at` 设定"20 分钟后"的一次性提醒。更重要的是，Cron 任务（尤其是 `isolated` 模式）可以指定更强的模型（比如 `opus`）、更高的思考级别，甚至执行完整的 `agentTurn` 去调用工具。

我原本想用 Heartbeat 做个每日日报，结果发现它没法保证在每天 23:30 准时发送，而且也没法强制用高思考级别去深度分析。最后只能老老实实用 Cron 配了个 `--cron "30 23 * * *" --tz "Asia/Shanghai"`，才搞定这个需求。

现在的策略很清晰了：日常监控、收件箱扫描这种"搭便车"的活交给 Heartbeat；而需要精确时间点、独立执行环境或者深度分析的任务，必须上 Cron。两者结合，才是最优解。

## 今天的收获

- `sessionTarget` 选错会直接污染主会话上下文，后台自动化任务必须强制使用 `isolated` 模式。
- Heartbeat 是"模糊感知"的批量检查机制，适合低成本监控；Cron 是"精确打击"的独立任务，适合定时执行和深度分析。
- Cron 任务若配置为 `systemEvent` + `main`，Agent 可能只"看到"消息而不执行实际操作，需改为 `agentTurn` + `isolated`。
- 时区配置 `--tz` 在 Cron 中至关重要，否则调度时间会随 Gateway 主机本地时区漂移。
- 最佳实践是 Heartbeat 处理日常感知，Cron 处理精确调度和独立任务，两者互补而非互斥。
