# AI Agent 自我进化系统设计指南

## Atoms

| 编号  | 类型 | 内容                                                                                                           | 原文定位                  |
| ----- | ---- | -------------------------------------------------------------------------------------------------------------- | ------------------------- |
| AE-01 | 事实 | 自我进化是 AI Agent 自主分析差距、设计方案、执行验证、反思迭代的闭环持续改进过程                               | 一、核心理念              |
| AE-02 | 判断 | 传统 Agent 是被动响应式，自我进化机制赋予其目标导向、自我优化、知识积累、资源意识的主动性                      | 为什么需要自我进化        |
| AE-03 | 经验 | 终极目标设定四原则：具体可衡量、有挑战性、可分解、有意义，作为进化的北极星                                     | 终极目标的设定            |
| AE-04 | 事实 | 基础版三阶段循环模型：规划 (Planner)→执行 (Executor)→跟踪 (Tracker)，输出计划/日志/反思报告                    | 基础版：三阶段循环模型    |
| AE-05 | 事实 | 规划阶段五任务：环境分析、差距识别、机会发现、方案设计、预算评估，输出结构化进化计划                           | 规划阶段                  |
| AE-06 | 事实 | 执行阶段五任务：步骤获取、条件检查、操作执行、结果记录、资源统计，高频运行每次只执行一个步骤                   | 执行阶段                  |
| AE-07 | 事实 | 跟踪阶段五任务：进度检查、预算审计、效果评估、决策制定 (继续/完成/放弃)、经验总结                              | 跟踪阶段                  |
| AE-08 | 事实 | OADA 闭环五阶段：Observe(观察)→Analyze(分析)→Decide(决策)→Act(执行)→Verify(验证)，实现全自动进化               | 进阶版：OADA 自主进化闭环 |
| AE-09 | 判断 | OADA 与三阶段模型区别：自动化程度 (全自动 vs 人工)、响应速度 (定时 vs 触发)、自我修改能力、数据驱动、风险控制  | OADA 与三阶段模型的区别   |
| AE-10 | 事实 | 自我分析器负责 Observe+Analyze：收集平台 API/日志/内容数据，生成洞察，按优先级输出可执行建议                   | 自我分析器的设计          |
| AE-11 | 事实 | 自我修改器负责 Act：修改发布计划/添加模板/更新策略，所有修改前自动备份带时间戳，支持一键回滚                   | 自我修改器的设计          |
| AE-12 | 事实 | 自动进化引擎协调 OADA 全流程：调用分析器→智能决策→调用修改器→验证效果→生成报告→准备下一周期                    | 自动进化引擎的设计        |
| AE-13 | 事实 | 任务状态流转：pending→in_progress→pending_review→completed 或 abandoned                                        | 任务状态流转              |
| AE-14 | 事实 | 步骤状态流转：pending→in_progress→completed 或 skipped                                                         | 步骤状态流转              |
| AE-15 | 经验 | 并发控制四原则：单任务原则 (同一时间仅一个活跃任务)、原子操作、锁机制、幂等设计                                | 并发控制                  |
| AE-16 | 判断 | 预算维度：时间预算 (建议 48-72 小时) 防拖延，Token 预算 (建议 5-10 万) 控成本，超限需预警/降级/记录原因        | 预算维度                  |
| AE-17 | 经验 | 调度策略：规划每日清晨 (06:00) 深思熟虑，执行每 2-4 小时推进进度，跟踪每日晚间 (22:00) 评估总结                | 调度策略                  |
| AE-18 | 经验 | 推荐使用 Cron 表达式实现定时调度，因其表达能力强、广泛支持、易于维护                                           | 调度实现                  |
| AE-19 | 事实 | 数据持久化结构：state.json(核心状态)、plans/(计划存档)、logs/(执行日志)、reflections/(反思报告)                | 存储结构                  |
| AE-20 | 经验 | 设计五原则：单一职责 (组件专一)、低耦合 (状态文件解耦)、可观测 (全行为追踪)、优雅降级、可扩展                  | 六、设计原则              |
| AE-21 | 判断 | 人机关系：Agent 处理日常维护/数据分析/内容微调/问题修复，人类负责策略决策/紧急情况/方向性变更                  | 增强而非替代              |
| AE-22 | 经验 | 保留人类干预权：随时中止、手动触发、回滚修改、调整参数、审核日志，系统需透明可解释                             | 保留人类干预权            |
| AE-23 | 经验 | 渐进式信任模式：初期全审批→稳定期低风险自动→成熟期大部分自动，信任通过持续良好表现赢得                         | 渐进式信任                |
| AE-24 | 事实 | 安全机制五层：备份策略 (修改前自动备份)、回滚能力 (一键/选择性/分类)、试运行模式 (dry-run)、限流保护、监控告警 | 八、安全机制              |
| AE-25 | 经验 | 实施四阶段：基本三阶段循环→预算和并发控制→日志和反思机制→调度策略和智能决策，渐进式完善                        | 渐进式实施                |
| AE-26 | 事实 | 短期优化 (1-2 周)：增加情感/竞争/时间分析维度，丰富修改能力，改进通知机制                                      | 短期优化                  |
| AE-27 | 事实 | 中期优化 (1 个月)：A/B 测试框架、目标追踪仪表板、智能内容生成                                                  | 中期优化                  |
| AE-28 | 事实 | 长期优化 (3 个月)：深度学习优化预测效果、跨平台/多 Agent 生态扩展、自我演进改进进化逻辑                        | 长期优化                  |
| AE-29 | 判断 | 两种模式适用场景：三阶段循环适合初期探索/复杂决策需人工配合，OADA 闭环适合成熟场景/常规优化全自动              | 两种模式                  |
