# KL11: Browser Relay 架构通过“本地中继服务器 + 扩展徽章状态机”实现安全可控的浏览器标签页共享控制

> 所属视角：P11-browser-relay
> 上层论点：Browser Relay 架构通过“本地中继服务器 + 扩展徽章状态机”实现安全可控的浏览器标签页共享控制，在避免独立浏览器实例资源开销的同时，填补了沙箱环境在“带登录态交互”场景的能力空白。

## 支撑论点

### 11.1: 本地中继服务器作为安全通信枢纽，建立了宿主浏览器与自动化任务间的可信数据通道

- 逻辑顺序：结构
- 引用 atoms: BR-01, BR-02
- 引用 groups: G11-A

本地中继服务器驻留在用户本地环境，充当了受控的代理层，既隔离了外部直接访问浏览器的风险，又允许自动化脚本通过标准协议注入指令，确保了通信链路的私密性与可控性。

### 11.2: 扩展徽章状态机精确管理标签页生命周期，解决了沙箱环境下带登录态交互的上下文保持难题

- 逻辑顺序：结构
- 引用 atoms: BR-03, BR-04
- 引用 groups: G11-B

通过在浏览器扩展中部署状态机，系统能够实时追踪并同步标签页的认证状态（Cookie/Session）和 DOM 上下文，使得自动化流程无需重新登录即可在现有用户会话中执行复杂操作。

### 11.3: 复用宿主浏览器实例而非启动独立无头浏览器，显著降低了系统资源开销并提升了执行效率

- 逻辑顺序：程度
- 引用 atoms: BR-05, BR-06
- 引用 groups: G11-C

该架构直接利用用户已打开的浏览器窗口和内存资源，避免了为每个任务单独启动 Chromium 实例所带来的巨大 CPU 和内存负担，实现了轻量级的高并发任务处理。

## 论点间关系

上述三个子论点按照“架构组成（通信枢纽 + 状态管理）”到“最终效益（资源优化）”的结构顺序展开：前两点阐述了实现安全可控共享的核心技术机制，第三点则说明了该机制带来的直接性能优势，共同支撑了上层关于填补能力空白且低开销的结论。
