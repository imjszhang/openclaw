# KL14: OpenClaw 外部集成需根据"自动化场景需求"在 CLI、HTTP API 与 WebSocket RPC 间进行差异化选型，以精准适配从本地脚本调用到复杂远程控制的不同业务形态。

> 所属视角：P14-external-integration
> 上层论点：系统架构以"本地优先网关"为核心枢纽，通过五层抽象连接多渠道与多工具，奠定安全路由基石

## 支撑论点

### 14.1: CLI 接口专为本地脚本调用与运维管理设计，提供直接操作 Gateway 核心命令的高效入口

- 逻辑顺序：结构
- 引用 atoms: AR-16, AR-04, AR-32
- 引用 groups: G15

CLI 作为最直接的交互层，封装了 `gateway`、`agent`、`send` 等核心命令，适用于本地自动化脚本触发、快速诊断（`doctor`）及配置管理，尤其适合在 Raspberry Pi 或 VPS 等无头环境中进行轻量级部署与维护。

### 14.2: HTTP API 服务于标准化的 Web 集成与状态监控，支持通过 RESTful 风格获取系统信息与触发简单事件

- 逻辑顺序：结构
- 引用 atoms: AR-11, AR-03, AR-08
- 引用 groups: G15

Gateway 内置的 HTTP 服务层提供了标准的接口能力，允许外部 Web 应用或监控系统在无需建立长连接的情况下，查询节点状态、资源使用情况（如 CPU/RAM 约束）或执行非实时的单向指令，满足松耦合的集成需求。

### 14.3: WebSocket RPC 是复杂远程控制与实时双向通信的首选，依托 Gateway 核心端口实现全双工消息路由与会话管理

- 逻辑顺序：程度
- 引用 atoms: AR-10, AR-11, CP-13, CP-02
- 引用 groups: G28

作为架构核心（端口 18789），WebSocket RPC 支持 Channels 到 Agents 的全链路数据流，能够处理复杂的 bindings 路由、实时事件广播及多会话并发控制，是构建实时伴侣应用、Live Canvas 及高级自动化流程的必要通道。

## 论点间关系

三个子论点按照**交互复杂度与实时性要求**的程度顺序排列：从本地单向脚本执行（CLI），到标准化的状态查询与简单触发（HTTP API），最后进阶至全双工、高并发的实时业务控制（WebSocket RPC），层层递进以覆盖不同深度的集成场景。
