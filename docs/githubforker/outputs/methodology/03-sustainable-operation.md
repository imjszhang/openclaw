# 第三章：保障体系持续运转

> 前置阅读：[第二章](02-architecture-design.md)
> 难度：中级

**文档化的增量流程、模板化的扩展入口和五项设计原则，共同保障拆解体系在持续新增中稳定运转。** 前两章解决了"为什么"和"怎么设计"，但知识库不是搭完架构就完事的——journal 会持续增长，新笔记不断涌入，拆解是一个永远不会一次性完成的过程。本章给出三个保障机制。

## 3.1 增量拆解流程文档化

**把"拿到一篇新 journal 后该做什么"写成一条可重复执行的流水线，每次新增沿流程走即可，无需重新摸索。**

```
新 journal 写完
    ↓
打散为 atoms（放入 analysis/atoms/当月目录）
    ↓
新 atoms 归入已有 groups 或创建新 group（更新 analysis/groups/）
    ↓
groups 有变动 → 重新审视 synthesis（更新 analysis/synthesis.md）
    ↓
synthesis 有变动 → 检查受影响的视角（更新 structure/Pxx-xxx/）
    ↓
视角的 tree 有变动 → 更新 validation 的缺口清单
    ↓
在各受影响文件的修订记录中留下变更条目
```

每一步的输入和输出都是明确的。拿到新笔记后不需要想"我该从哪开始"——沿着这条路径往下走就行。

文档化还有一个好处：当你隔了一段时间回来继续拆解，不用重新理解整个体系的运作逻辑，看流程就知道上次停在哪、下一步该做什么。

## 3.2 视角模板化

**提供 `_template/` 目录作为新视角的快速起步入口，复制改名即可获得完整骨架，既降低新建成本又保障格式一致。**

建新视角是拆解过程中最"重"的操作——需要新建目录、创建 SCQA、搭建 tree 骨架、准备 validation。如果每次都从空白开始，成本高且容易遗漏。模板的具体结构如下：

```
structure/_template/
├── scqa.md        SCQA 骨架（四要素占位 + 读者画像模板）
├── tree/
│   └── README.md  塔尖 + Key Line 表格模板
└── validation.md  MECE 检查 + 逻辑链检查 + 缺口清单模板
```

新建视角时，复制 `_template/` 改个名就能得到完整的骨架文件。填入内容即可，不需要从头回忆"该有哪些文件、每个文件什么格式"。

模板化还保障了格式一致：所有视角的目录结构相同，文件命名相同，表格格式相同。这让跨视角的比较和审查变得容易——你一眼就能看出哪个视角还缺 validation，哪个视角的 tree 还没展开。

## 3.3 五项设计原则

**职责分离、扩展性优先、共享与隔离、增量友好、可追溯——这五条原则既是前两章架构决策的事后总结，也是未来遇到新的组织困惑时的决策指南。**

**职责分离。** 每一层、每一个区域只做一件事。journal 只管记录，analysis 只管拆解，structure 只管组织，outputs 只管表达。职责混合是复杂度失控的根源。

**扩展性优先。** 架构设计优先考虑"加新东西时是否简单"。新 journal → 新 atoms → 影响已有 groups → 自然传导。不需要重构现有结构就能容纳新内容。

**共享与隔离。** analysis 层跨视角共享（一条信息可以同时支撑多个金字塔），structure 层按视角隔离（不同读者看到不同的组织方式）。共享减少重复，隔离保护独立性。

**增量友好。** 所有产物都支持增量更新，不需要"推倒重来"。新 journal 只需增量走一遍流程，已有产物只在受影响时才更新。

**可追溯。** 每个论点能追溯到 atoms，每个 atom 能追溯到 journal 原文。修订记录保留每次变更的时间和原因。当你质疑某个结论时，顺着链路就能找到它的依据。

---

当遇到架构选择上的犹豫时，回到这五条原则来验证：如果某个方案违反了其中任何一条，大概率不是好选择。
