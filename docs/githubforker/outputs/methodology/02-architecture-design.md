# 第二章：架构设计的两条主轴

> 前置阅读：[第一章](01-why-restructure.md)
> 难度：中级

上一章说清了"为什么要重组"和"用什么方法"。本章进入具体设计：宏观上怎么分层？中观上 pyramid 内部怎么设计？微观上每个产物怎么存放？

这三个层次的设计由两条主轴贯穿：**三层分离**和**增长驱动的组织策略**。

## 2.1 输入→处理→输出：三层各管一件事

**整个知识库分为 journal（输入）、pyramid（处理）、outputs（输出）三层，每层有且只有一个职责，层间不混合。**

```
journal/（输入层）→ pyramid/（处理层）→ outputs/（输出层）
```

**journal/ 是输入层**——原始学习日志保持不动，按日期归档。它的职责是忠实记录，永远不去修改已经写下的内容。哪天写的就放在哪天的目录下，这层只管"存"。

**pyramid/ 是处理层**——用金字塔原理对素材进行结构化拆解。打散、分组、归纳、验证，都在这层完成。这层只管"理"。

**outputs/ 是输出层**——面向读者的最终产出。教程、文章、博客、手册，任何形式的"给别人看的东西"都在这层。这层只管"表达"。

**为什么教程不应该跟笔记放在一起？** 一个常见的直觉是把教程直接放在顶层——毕竟它也是你写的内容。但教程是 pyramid 拆解的产出物，它的内容源自对 journal 的结构化处理。如果把它和 journal、pyramid 同级存放，三层的职责边界就会模糊。

## 2.2 分析共享与表达隔离——pyramid 内部的双轨设计

**pyramid 内部分为 analysis（自下而上，所有视角共享）和 structure（自上而下，按视角独立），因为"从素材中发现了什么"和"要传达什么"是两件不同的事。**

pyramid 因此分为两个区域：

```
pyramid/
├── analysis/（自下而上：所有视角共享的素材池）
│   ├── atoms/    打散的信息单元
│   ├── groups/   按逻辑分组 + 观点句
│   └── synthesis.md  收敛为顶层候选观点
│
└── structure/（自上而下：每个视角独立的组织方案）
    ├── P01-xxx/  视角一（scqa + tree + validation）
    ├── P02-xxx/  视角二
    └── ...
```

**analysis 是共享的。** 不管你将来要写方法论文章还是技术教程，素材拆解的结果（atoms、groups、synthesis）都是通用的。一条信息单元可能同时支撑多个视角。

**structure 是按视角隔离的。** 不同的读者群体会产生完全不同的金字塔结构——面向方法论学习者的 P01 和面向技术操作者的 P02，虽然共享同一个素材池，但 SCQA 引入、Key Line 编排、论点深度都不一样。

**关键区分：synthesis 的终点不等于 SCQA 的起点。** synthesis 回答的是"我从素材中发现了什么规律"（分析视角），而 SCQA 回答的是"我要对这批读者传达什么"（表达视角）。前者是归纳的终点，后者是表达的起点。它们之间有关联，但不是简单的复制。

**按视角分目录，而非按产物类型分。** structure 下面不是 `scqa/`、`tree/`、`validation/` 这样按文件类型分的目录，而是 `P01-xxx/`、`P02-xxx/` 这样按视角分的目录。因为同一种产物（比如 scqa.md）在不同视角下内容完全不同，它们不应该放在一起。

## 2.3 增长驱动的组织策略

**产物是否随素材规模增长决定其组织方式：会膨胀的用目录拆分，固定的用单文件保内聚。** 架构搭好后，每个具体产物——atoms、scqa、tree——到底是用一个文件还是一个目录，由这条准则统一裁定。

### 会增长的，用目录

**atoms 按月分子目录。** 每拆解一篇新 journal 就会产生新的 atoms。如果全部扁平堆在一个目录下，几十篇 journal 之后文件列表会变得不可管理。按月分目录（`2026-01/`、`2026-02/`）跟 journal 的日期结构呼应，自然限制了单个目录的文件数量。

**tree 从单文件升级为目录。** 金字塔展开后，一个 Key Line 下面可能有三四个子论点，每个子论点引用多个 atoms。如果全写在一个 tree.md 里，文件会很快膨胀到难以阅读。每个 Key Line 独立一个文件，保持了单文件的可读性。

### 不会增长的，用单文件

**SCQA 保持单文件。** S、C、Q、A 四要素加上读者画像构成一条紧密的逻辑链——情境引出冲突，冲突引出疑问，疑问引出答案。它们的价值在于"一眼扫完就能判断逻辑是否通顺"。如果拆成四个文件，验证就要在文件之间来回跳转，反而破坏了整体验证的价值。

**validation 保持单文件。** MECE 检查、逻辑链检查、缺口清单——这些是对整棵金字塔的系统性体检，内容固定，不随素材增长。

这条原则的好处是：你不需要预先规划所有目录结构，只需要在产物开始膨胀时按规则拆分即可。
